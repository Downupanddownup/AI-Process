# V5 技术方案补充说明

## 1. 这一轮新增重点

你这次提的问题很实际，主要集中在四件事：

1. 前面提到的技术名词到底是什么意思
2. 这个程序应该如何快速启动
3. 这个程序如何始终保持在其他窗口前面
4. 这个程序最小化后如何隐藏到右下角托盘

这几个问题都属于“桌面工具的使用方式设计”，而且很关键。


## 2. 前面提到的技术名词，简单解释一下

### 2.1 Tauri 是什么

你可以把它理解成：

`一个用来做桌面软件的轻量级外壳`

它的作用是：

1. 让网页界面变成 Windows 桌面程序
2. 让程序可以访问本地文件、剪贴板、窗口控制、托盘等系统能力

为什么我提它：

因为它比 Electron 更轻，更适合你这种小工具。


### 2.2 Electron 是什么

它也是做桌面软件的。

很多常见桌面软件都用它做。

但它的问题是：

`通常偏重`

对于你这种小工具来说，能做，但没必要优先选它。


### 2.3 Vanilla TypeScript 是什么

这个名字别被吓到。

它的意思其实很简单：

`不用 React、Vue 这些大框架，直接用原生网页方式写界面`

好处是：

1. 更轻
2. 文件更少
3. 更容易控制
4. 适合按钮不多、界面不复杂的小工具


### 2.4 JSON 配置文件是什么

你可以把它理解成：

`一个专门放提示词和界面文案的文本配置文件`

比如：

1. 需求提示词写在里面
2. 回复提示词写在里面
3. 文件关系说明写在里面
4. 按钮名字也可以写在里面

以后你想改文案，直接改这个文件，重启程序就生效。


## 3. 关于程序如何启动

这是第一版必须认真设计的点。

你这个工具本质上不是“先打开再慢慢用”的大程序，而是：

`需要随时呼出、随时隐藏的小面板`

所以启动方式我建议分成两层：

### 3.1 第一层：正常启动

程序双击启动后：

1. 程序进入系统托盘
2. 主窗口默认显示在屏幕顶部
3. 如果你关闭主窗口，不代表彻底退出程序，而是隐藏回托盘

这意味着它更像一个常驻工具，而不是每次完整退出。


### 3.2 第二层：快捷键呼出

这是更符合你使用习惯的方式。

建议支持一个全局快捷键，例如：

1. `Alt + Space`
2. `Ctrl + Shift + Space`
3. 或你自定义的组合键

作用是：

1. 如果窗口当前隐藏，则呼出窗口
2. 如果窗口当前已显示，则将其激活到最前面
3. 光标自动聚焦到路径输入框或主操作区

这个能力非常重要，因为它决定了这个工具能不能像 AutoHotkey 那种小工具一样顺手。


## 4. 我建议的程序运行模式

第一版不适合做成“点开一次，用完关闭”的模式。

更适合做成：

`托盘常驻 + 快捷键呼出 + 手动隐藏`

理由：

1. 符合高频使用场景
2. 启动一次后就可以反复调用
3. 避免每次都重新完整打开
4. 更接近你提到的 AutoHotkey 小工具体验


## 5. 关于始终保持在最前端

这个点很关键，而且是明确的桌面窗口能力。

你要的是：

`这个小窗口在显示状态下，始终位于其他普通窗口前面`

也就是说：

1. 你点 IDE
2. 你点浏览器
3. 你点聊天窗口

这个工具窗口依然不会被盖住。

这在实现上通常叫：

`Always On Top`

也就是“窗口置顶”。

### 我建议的行为

1. 窗口显示时，默认置顶
2. 点击其他窗口时，它不被普通窗口覆盖
3. 除非你主动最小化或隐藏它

这符合你对“小面板始终悬浮”的要求。


## 6. 这里要补一个交互边界

虽然你要求它始终置顶，但如果完全一直挡着，也可能影响操作。

所以比较合理的方式是：

1. 平时通过快捷键呼出
2. 呼出后置顶显示
3. 用完后你主动最小化或隐藏

而不是让它永远常驻屏幕中间不动。

也就是说：

`置顶` 和 `可快速隐藏` 必须配套设计。


## 7. 关于最小化行为

你这次明确提出了一个非常重要的要求：

`最小化时，不是缩到任务栏，而是隐藏到桌面右下角托盘`

这个方向我认同，而且很适合这种小工具。

建议行为如下：

### 7.1 点击最小化按钮时

执行：

1. 主窗口隐藏
2. 程序继续驻留在系统托盘
3. 任务栏上不保留普通窗口占位


### 7.2 托盘图标右键菜单

建议最少包含：

1. `显示主窗口`
2. `退出程序`

如果要再加一个，可以加：

1. `重新加载配置`

但这个不是第一版必须项。


### 7.3 双击托盘图标

建议行为：

1. 恢复显示主窗口
2. 并自动置顶到最前面


## 8. 关闭按钮的行为也要提前定

这个点你虽然没直接问，但和托盘强相关，所以必须一起定。

需要明确：

`点击窗口右上角关闭按钮时，到底是退出程序，还是隐藏到托盘`

我建议第一版采用：

1. 点击关闭按钮：隐藏到托盘
2. 真正退出程序：通过托盘右键菜单里的 `退出程序`

理由：

1. 更符合常驻工具逻辑
2. 避免你误关后又得重新启动
3. 和很多输入法、截图工具、快捷工具的行为一致


## 9. 启动与隐藏的完整体验建议

我建议第一版把整个行为统一成下面这套：

### 启动时

1. 程序启动
2. 进入托盘
3. 主窗口显示
4. 主窗口默认置顶

### 使用中

1. 你通过快捷键呼出窗口
2. 窗口始终在最前面
3. 操作完成后，你可以点最小化

### 最小化时

1. 主窗口隐藏到托盘
2. 不占任务栏主位置

### 再次使用时

1. 通过快捷键重新呼出
2. 或点击托盘图标恢复

### 退出时

1. 通过托盘菜单点 `退出程序`


## 10. 这会影响技术选型吗

不会，反而进一步说明：

`Tauri 这种桌面壳子是合适的`

因为你现在需要的不只是一个页面，而是明确的桌面能力：

1. 全局快捷键
2. 窗口置顶
3. 最小化隐藏到托盘
4. 托盘菜单
5. 本地文件操作
6. 剪贴板能力

这些都属于桌面应用能力，不是纯网页能自然解决的。


## 11. 对第一版技术路线的修正

相比上一版，这一版技术路线可以更明确一点：

`Tauri + 原生轻界面 + JSON 配置 + 托盘常驻`

也就是说，第一版不仅是一个桌面程序，还是一个：

`带托盘、带快捷键、带置顶能力的小型常驻工具`


## 12. 第一版实施计划补充

在前一版实施计划基础上，需要增加两个阶段。

### 新增阶段 A：窗口行为实现

目标：

1. 主窗口默认置顶
2. 支持最小化隐藏到托盘
3. 支持从托盘恢复窗口
4. 支持关闭按钮隐藏到托盘

验收：

1. 点击最小化后，窗口不出现在任务栏主区域
2. 托盘图标可恢复窗口
3. 恢复后窗口仍然置顶


### 新增阶段 B：快捷键呼出

目标：

1. 注册全局快捷键
2. 支持隐藏状态下唤醒窗口
3. 支持显示状态下激活窗口

验收：

1. 无论当前焦点在哪个程序，快捷键都能呼出此工具


## 13. 现在最适合确认的几个实现细节

这轮之后，我认为最值得尽快确定的是：

1. 全局快捷键最终用哪个组合
2. 程序启动时是默认显示窗口，还是只进入托盘
3. 点击关闭按钮时，是隐藏到托盘，还是直接退出
4. 托盘菜单是否只保留“显示”和“退出”两个选项


## 14. 结论

这一轮之后，技术方案比前面更完整了。

现在这个工具不只是“轻量配置驱动的小程序”，而是一个明确的 Windows 常驻快捷工具。

它的第一版目标应该是：

1. 轻量
2. 置顶
3. 托盘常驻
4. 快捷键呼出
5. 配置集中维护

如果按这个方向实现，整个工具的使用体验会更接近你真正想要的那种感觉，而不是一个普通小窗口程序。
