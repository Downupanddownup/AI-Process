# 确认理解 + 对当前 Summary.md 的价值分析 v3

## 一、确认你的四点答复

1. **分类数量**：6 类 OK，确认空转的边界是「没有新的信息补充，只是确认」。
2. **极端失控**：它是纠偏轮次的一个特化，不是独立大类。最终分类输出里可以保留「极端失控」这个标签，但逻辑上它属于纠偏的子集。
3. **时间来源**：优先用 `log.jsonl`，没有日志时回退到文件元数据。
4. **两轮分类是 AI 执行步骤**：不是输出两个文件，而是让 AI 内部先做基础 6 类判断，再对纠偏轮次做 extreme 特化判断，最终只输出一个结果。

下面我基于这四点，先分析一下当前 `Summary.md` 里哪些内容有价值、哪些偏低价值，然后再给出一个统一的最终输出结构。

## 二、当前 Summary.md 的内容价值分析

我之前看了 `需求/历史财富/.aiprocess/Summary.md`，全长 290 行。按我的判断，内容价值分层如下：

### 2.1 高价值内容（应保留并强化）

| 内容 | 价值点 |
|---|---|
| **1.1 内容摘要** | 用 2~4 句话概括主题做了什么，这是第一眼价值。 |
| **复杂程度 / 顺畅程度** | 虽然现在是主观三档，但可以量化后更具体，比如 "9 轮中 2 轮纠偏、1 轮失控"。 |
| **阶段耗时（讨论/执行/结果微调）** | 能看出时间花在哪一阶段，是后续统计的基础。 |
| **嵌套子主题耗时表** | 对复杂主题很重要，能看出哪个子主题拖了时间。 |
| **3. 预期之外的问题** | 这是经验沉淀的核心，表格形式很好。 |
| **4. 经验教训** | 跨主题复用的价值所在，应保留。 |

### 2.2 低价值或可考虑精简的内容

| 内容 | 问题 |
|---|---|
| **5. 目录结构（第 61~290 行）** | 占了全文件 80% 的篇幅，只是文件树 + 创建时间。它确实能帮助快速了解进程，但信息量密度极低，而且对后续统计几乎没有结构化价值。 |
| **基础信息里的「文件总数 / 字符总量」** | 有参考价值，但单独列出来意义不大，更适合作为指标卡片里的一行。 |
| **「预期之外的问题」和「经验教训」偶尔重叠** | 有些经验教训几乎就是对问题的重复描述，没有真正上升到可跨主题复用的层面。 |
| **标签** | 有归纳作用，但现在是由 Agent 自由提取，标准不统一，后续统计时难以聚合。 |

### 2.3 当前文件最大的问题

1. **太长**：目录结构把核心内容淹没了。
2. **结构自由**：虽然是按模板输出，但 Agent 仍然有很大的发挥空间，导致不同主题的 Summary.md 质量不稳定。
3. **缺少轮次视角**：虽然文件树按时间排列，但没有明确把「一轮一轮」拎出来，也就没法回答「哪一轮开始跑偏」「失控是怎么升级的」这类问题。
4. **缺少量化连接**：复杂/顺畅程度是文字三档，没有和具体的轮次分类数据挂钩。

## 三、新 Summary 应该长什么样（一个最终文件）

按你的要求，最终只输出一个结果。我建议这个结果就是一份 **结构化的 JSON 文件**，同时可以辅助生成一份精简的人类可读 Markdown（可选）。核心原则：

- **轮次是骨架**：先有时间轴，再有规律判断。
- **高价值内容前置**：摘要、指标、问题、经验教训放在最前面。
- **低价值内容后置或删除**：目录结构可以大幅压缩，只保留关键节点。

## 四、AI 执行流程（内部两步，输出一个 JSON）

按你的意思，AI 内部执行两步，但只输出一个最终结果：

**第一步：读取基础轮次文件**
- 输入：本地脚本提取的 `{themePath}\.aiprocess\summary_rounds_base.json`
- 内容：每轮的人/AI 文件路径、字数、时间、阶段

**第二步：AI 做两轮判断（内部）**
1. **基础分类**：给每轮打上 6 类之一。
2. **极端失控特化**：在已标为「纠偏拉回」的轮次中，判断哪些是「极端失控」。

**第三步：输出最终 JSON**
- 包含 `overview`、`metrics`、`timeline`、`patterns`、`issues`、`lessons` 等。
