## 三、关键字段说明

### 3.1 `rounds.items` 单轮字段

| 字段 | 含义 | 来源 |
|---|---|---|
| `roundIndex` | 轮次编号，从 0 开始 | 脚本 |
| `stage` | 讨论 / 执行 / 结果微调 | 脚本 |
| `humanFile` / `aiFile` | 人输入文件 / AI 输出文件 | 脚本 |
| `startTime` / `endTime` | 轮次起止时间 | 脚本（优先 log， fallback 文件时间） |
| `duration` | 耗时，含分钟数和人类可读显示 | 脚本计算 |
| `humanSummary` | 本轮用户输入的核心内容摘要 | AI |
| `humanRequirementCount` | 本轮用户提出了几个需求点/要求 | AI |
| `aiSummary` | 本轮 AI 回复的核心内容摘要 | AI |
| `category` | 6 类之一 | AI 第一轮 |
| `isExtreme` | 是否属于极端失控 | AI 第二轮 |

### 3.2 `rounds.phases` 跨轮汇总

| 字段 | 含义 | 来源 |
|---|---|---|
| `name` | 阶段名称，如「方向探索」「方案收敛」 | AI |
| `roundRange` | 轮次范围，如 "R0 ~ R2" | AI 或脚本 |
| `summary` | 该阶段在做什么 | AI |
| `categories` | 该阶段主要包含哪些轮次类型 | AI |

### 3.3 时间显示优化

你提到现在 "0天 9小时 24分钟" 太繁琐。我建议统一用以下规则：

```
< 1 分钟："<1分钟"
< 1 小时："25分钟"
< 1 天："9小时 24分钟"
≥ 1 天："1天 3小时 15分钟"
```

同时保留原始分钟数 `duration.minutes` 和 `overview.totalTime.minutes`，方便 HTML 做图表和排序。显示字符串只用于人类阅读。

### 3.4 性能指标说明

| 指标 | 计算方式 |
|---|---|
| `intentMatchRate` | `(总字符数 - 纠偏轮次人输入字符数 - 极端失控轮次人输入字符数) / 总字符数` |
| `correctionTextRatio` | `纠偏轮次人输入字符数 / 总字符数` |
| `extremeTextRatio` | `极端失控轮次人输入字符数 / 总字符数` |

> 这里只统计「人输入」的字符数，因为纠偏成本主要体现在用户花了多少文字去纠正 AI。

### 3.5 `analysis` 替代「预期之外的问题」

按你的意思，把原来的 `issues` 拆成两个更有针对性的部分：

- **`correctionAnalysis`**：对所有纠偏轮次的分组分析。每组说明：
  - 涉及哪些轮次
  - 纠偏内容摘要
  - 根因
  - 影响
  - 可复用经验

- **`extremeAnalysis`**：如果存在极端失控轮，单独分析每个失控轮：
  - 触发点
  - 升级路径
  - 解决方式
  - 避免方法

如果某主题没有纠偏/失控，对应数组为空。

## 五、HTML 展示页面（后续开发）

HTML 程序读取同目录下的 JSON 文件，建议展示布局：

- **顶部总览卡片**：摘要、总耗时、复杂程度、协作顺畅度、关键指标。
- **指标区**：意图匹配率进度条、纠偏率、失控率、轮次分布饼图。
- **阶段时间轴**：用不同颜色区分阶段，横向或纵向时间轴。
- **轮次明细表**：可折叠，显示每轮摘要、类型、耗时。
- **纠偏/失控分析卡片**：突出显示， expandable。
- **子主题耗时条形图**。

这个 HTML 程序可以做得比较轻量，纯静态 HTML/JS，不需要后端。

## 六、还需要你确认的问题

1. **`humanRequirementCount` 的具体含义**：是指用户本轮提出的「独立需求点数量」，还是「段落/条目数量」？例如用户写了一段话里包含三个要求，是算 3 还是算 1？

2. **`rounds.phases` 的命名**：阶段名称是让 AI 自由命名（如「方向探索」「方案收敛」），还是按固定模板（如「讨论期」「细化期」「纠偏期」「收尾期」）？

3. **`analysis.correctionAnalysis` 的分组方式**：
   - 是按「同一类根因」自动合并（如所有"没看代码"的纠偏合并成一条）？
   - 还是每轮纠偏都单独分析？
   - 我倾向于按根因合并，这样经验更精炼。

4. **是否保留 `keyMilestones`**？还是直接由 HTML 从 `rounds.phases` 推导关键节点？
