# 修正理解：Markdown/HTML 是程序功能，不是 AI 输出 v7

## 一、修正 v6 里的误解

v6 里我把 Markdown 和 HTML 写成了「脚本渲染产物」，这个理解有误。重新确认：

> **Markdown 生成 和 HTML 展示是 AI Process 的功能模块，由 AI Agent 一次性写出程序代码后稳定运行，后续不再依赖 AI。**

也就是说：

- AI 当前只需要生成 `Summary.json`。
- Markdown 生成器和 HTML 查看器是后续要开发的程序功能，属于 AI Process 这个工具本身。
- 开发完成后，用户在「经验总结」窗口点击按钮，程序就读取 JSON 生成/展示结果。

## 二、四个文档的职责与稳定性

| 文档 | 类型 | 是否变化 | 说明 |
|---|---|---|---|
| `summary_input.json` | 基础素材 | **每次总结都变化** | 由脚本根据实际主题数据生成 |
| `summary_input_guide.md` | 素材说明 | **固定不变** | 解释素材字段含义 |
| `summary_template.json` | JSON 模板 | **固定不变** | 规定输出 JSON 的结构 |
| `summary_rules.md` | 生成规则 | **固定不变** | 规定 AI 如何分类、摘要、分析 |

只有 `summary_input.json` 是动态的，其余三份都是稳定的规则文档，可以放在 AI Process 的模板目录里长期使用。

## 三、AI 自检规则细化

你要求 AI 生成 JSON 后必须自己读一遍，验证格式和规则。我把自检流程细化如下：

### 3.1 格式自检

AI 必须验证 JSON 可以正常解析，没有语法错误。

### 3.2 字段完整性自检

逐项检查以下必填字段是否存在且非空：

- `themeName`、`themePath`、`generatedAt`
- `rounds.total`、`rounds.items`、`rounds.phases`
- `overview.abstract`、`overview.complexity`、`overview.complexityReason`、`overview.collaboration`、`overview.collaborationReason`、`overview.totalTime`、`overview.stageTime`
- `metrics.totalRounds`、`metrics.totalTimeMinutes`、`metrics.totalHumanChars`、`metrics.totalAiChars`、`metrics.byRound`、`metrics.byTime`、`metrics.byChars`
- `analysis.correctionAnalysis.perRound`、`analysis.correctionAnalysis.summary`
- `analysis.extremeAnalysis.perRound`、`analysis.extremeAnalysis.summary`

### 3.3 数值一致性自检

- `rounds.total` 必须等于 `rounds.items` 的长度。
- `rounds.items` 中的 `roundIndex` 必须从 0 开始连续递增。
- `metrics.byRound` 中各类别数量之和必须等于 `metrics.totalRounds`。
- `metrics.byTime.smoothMinutes + correctionMinutes + extremeMinutes` 必须等于 `metrics.totalTimeMinutes`（允许 1 分钟以内误差）。
- `metrics.byChars.smoothHumanChars + correctionHumanChars + extremeHumanChars` 必须等于 `metrics.totalHumanChars`。
- 所有比率字段必须在 0~1 之间。
- `byRound.intentMatchRate + byRound.correctionRate` 应该等于 1（因为分类互斥）。

### 3.4 分类规则自检

- 每个 `rounds.items[].category` 必须是以下 6 类之一：正常推进、细化完善、探索发散、纠偏拉回、极端失控、确认空转。
- `isExtreme` 为 `true` 的轮次，`category` 必须是「纠偏拉回」。
- 「极端失控」轮次的数量必须等于 `analysis.extremeAnalysis.perRound` 的长度。
- 「纠偏拉回」轮次的数量必须等于 `analysis.correctionAnalysis.perRound` 的长度。

### 3.5 分析内容自检

- `analysis.correctionAnalysis.perRound` 必须逐轮列出，不允许把多个轮次合并成一条。
- `analysis.extremeAnalysis.perRound` 同样必须逐轮列出。
- 每条分析必须包含 `roundIndex`、`category`、`summary`、`rootCause`、`impact`、`lesson`。
- `lesson` 不能简单重复 `summary`，必须更抽象、可跨主题复用。
- `summary`（最终汇总）必须在 `perRound` 全部列完后再生成，不能覆盖细节。

### 3.6 阶段划分自检

- `rounds.phases` 中每个阶段的 `name` 必须来自标准集合：需求探索期、方案讨论期、方案细化期、执行实施期、纠偏处理期、结果微调期、确认收尾期。
- 每个阶段的 `roundRange` 必须是有效的连续轮次范围，如 "R0 ~ R2"。
- 所有阶段覆盖的轮次范围合起来必须等于全部轮次，不能遗漏或重复。

## 四、关于 Markdown/HTML 功能实现的两种方式

你提到两种实现方式，我整理一下各自特点：

### 方式 A：文件落地

- 每次生成总结后，程序同时在主题目录下生成 `Summary.json` 和 `Summary.md`。
- HTML 页面可以放在 AI Process 程序目录下，但可以通过一个快捷方式在主题目录生成 `Summary.html` 调用它。
- 优点：文件可以直接查看、分享、版本控制。
- 缺点：每次总结都要写两份文件，主题目录会更拥挤。

### 方式 B：按需生成/展示

- 只保存 `Summary.json` 在主题目录下。
- Markdown 和 HTML 由程序在点击按钮时实时渲染/展示，不持久化到主题目录。
- HTML 页面代码只有一份，通过 URL 参数或文件路径加载不同的 JSON 数据。
- 优点：主题目录干净，JSON 是唯一真相源。
- 缺点：想看 Markdown 需要点击按钮生成，不能直接打开文件。

我建议 **方式 B**，原因：
1. JSON 是计算文件，Markdown 和 HTML 只是不同视图，不需要各存一份。
2. 后续如果要改展示样式，只需要改程序，不用改每个主题目录下的文件。
3. 主题目录不会越来越臃肿。

但方式 A 也有它的好处，特别是如果你希望直接打开 `Summary.md` 查看历史总结。你倾向哪种？

## 六、当前最重要的：JSON 文件怎么生成

1. **本地脚本读取主题目录**，按 TXT→MD 规则提取轮次，计算时间、字数、阶段。
2. **生成 `summary_input.json`** 放到 `.aiprocess\_tmp\`。
3. **AI Agent 读取** `summary_input.json` + 三份稳定规则文档。
4. **AI 生成 `Summary.json`** 并自检。
5. **程序后续读取 `Summary.json`** 做 Markdown/HTML 展示。
