## 二、性能指标定义（三层维度）

### 2.1 轮次占比

按轮次数量计算：

```
意图匹配率（按轮次）= (正常推进 + 细化完善 + 确认空转) / 总轮次
纠偏率（按轮次）= (纠偏拉回 + 极端失控) / 总轮次
失控率（按轮次）= 极端失控 / 总轮次
```

> 确认空转虽然信息增量低，但它本身不是 AI 理解错误，所以算入顺畅部分。

### 2.2 时间占比

按各轮耗时计算：

```
顺畅时间 = 正常推进 + 细化完善 + 确认空转 的耗时总和
纠偏时间 = 纠偏拉回 + 极端失控 的耗时总和

意图匹配率（按时间）= 顺畅时间 / 总时间
纠偏率（按时间）= 纠偏时间 / 总时间
失控率（按时间）= 极端失控耗时 / 总时间
```

### 2.3 字数占比

按用户输入字数计算（只统计人输入，因为纠偏成本主要体现在用户纠正 AI 上）：

```
顺畅字数 = 正常推进 + 细化完善 + 确认空转 的人输入字数
纠偏字数 = 纠偏拉回 + 极端失控 的人输入字数

意图匹配率（按字数）= 顺畅字数 / 总人输入字数
纠偏率（按字数）= 纠偏字数 / 总人输入字数
失控率（按字数）= 极端失控人输入字数 / 总人输入字数
```

> 字数占比作为参考，因为语音速记和打字的信息密度不同。

## 三、标准阶段名称与判定规则

轮次章节（`rounds.phases`）由 AI 按规则自动划分，但名称必须从下面这个固定集合里选：

| 阶段名称 | 判定规则 | 典型轮次类型 |
|---|---|---|
| **需求探索期** | 用户还在描述问题、提出方向，AI 提供多种可能方案，尚未收敛 | 探索发散、正常推进 |
| **方案讨论期** | 围绕某个或某几个具体方案展开讨论，逐步收敛 | 探索发散、正常推进、纠偏拉回 |
| **方案细化期** | 方向已确定，用户回答 AI 的确认性问题、填充实现细节 | 细化完善、正常推进、确认空转 |
| **执行实施期** | 进入具体实现，讨论代码、文件、模块修改 | 细化完善、正常推进、纠偏拉回 |
| **纠偏处理期** | 连续多轮出现 AI 理解偏差或实现错误，用户持续纠正 | 纠偏拉回、极端失控 |
| **结果微调期** | 功能基本完成后，针对细节、bug、体验做微调 | 细化完善、纠偏拉回、确认空转 |
| **确认收尾期** | 主题接近尾声，用户确认无问题、收尾 | 确认空转、正常推进 |

### 3.1 阶段划分约束

1. **连续原则**：一个阶段至少包含 2 个连续轮次。单个特殊轮次不单独成阶段，归入相邻阶段。
2. **互斥原则**：每个轮次只属于一个阶段。
3. **顺序灵活原则**：一个主题不一定经历所有阶段，顺序也可以回退（如执行期后回到纠偏期）。
4. **纠偏处理期特殊规则**：只有当连续 2 轮及以上都是「纠偏拉回」或「极端失控」时，才单独划分出「纠偏处理期」。否则单轮纠偏归入相邻阶段。

## 四、Correction Analysis 逐轮分析 + 最终汇总

按你的要求，不自动合并。结构改为：

```json
"correctionAnalysis": {
  "perRound": [
    {
      "roundIndex": 2,
      "category": "纠偏拉回",
      "summary": "AI 建议用户手动创建回复文件，用户指出应由程序自动生成。",
      "rootCause": "AI 对现有代码流程理解不足，按通用经验推断。",
      "impact": "浪费 1 个轮次，导致用户需要纠正基础流程。",
      "lesson": "涉及文件创建类功能时，应先确认当前代码是否已有自动化逻辑。"
    }
  ],
  "summary": {
    "totalCorrectionRounds": 3,
    "totalExtremeRounds": 1,
    "mainPattern": "主要问题集中在 AI 未先阅读代码就给出实现方案。",
    "keyLesson": "大需求启动前，AI 应先梳理项目代码和现有约定，把能由代码回答的问题内部消化。"
  }
}
```

### 4.1 每轮纠偏分析的必填字段

| 字段 | 含义 | 约束 |
|---|---|---|
| `roundIndex` | 对应轮次 | 必须存在 |
| `category` | 纠偏拉回 / 极端失控 | 从已分类结果中继承 |
| `summary` | 纠偏内容摘要 | 1~2 句话，讲清 AI 哪里错了、用户怎么纠正 |
| `rootCause` | 偏差根因 | 从 {漏读, 误读, 幻觉, 模型惯性, 环境差异, 未读代码} 中选一或自由描述 |
| `impact` | 造成的影响 | 浪费轮次、返工、方向摇摆等 |
| `lesson` | 可复用经验 | 必须比 summary 更抽象，能跨主题复用 |

### 4.2 最终汇总约束

- 必须等 `perRound` 全部列完后，再做汇总。
- 汇总里要指出主要模式和最关键的一条经验。
- 不允许汇总覆盖或替代 `perRound` 的细节。

## 七、每个模块的输入、输出与约束汇总

| 模块 | 输入 | 输出 | 关键约束 |
|---|---|---|---|
| **轮次提取器** | 主题目录下文件 | `rounds.items`（无 category） | 按 TXT→MD 规则配对；时间优先 log，fallback 文件时间；计算每轮字数 |
| **轮次分类器** | `rounds.items` + 文件内容 | `category`、`isExtreme` | 6 类标准；极端失控只在纠偏轮中判定 |
| **阶段划分器** | 已分类的 `rounds.items` | `rounds.phases` | 名称从标准集合选；连续 2 轮以上纠偏才成独立纠偏期 |
| **摘要生成器** | 文件内容 + 分类结果 | `humanSummary`、`aiSummary`、`overview.abstract` | 摘要必须客观、简短 |
| **指标计算器** | 分类结果 + 时间 + 字数 | `metrics` | 输出轮次、时间、字数三套指标 |
| **纠偏分析器** | 所有纠偏/失控轮次 | `analysis.correctionAnalysis`、`analysis.extremeAnalysis` | 每轮单独分析，最后汇总；不允许合并偷懒 |
