### 3.2 `summary_input_guide.md` —— 素材说明文件

告诉 AI 每个字段是什么、怎么来的、生成时怎么用。

```markdown
# Summary 素材说明

## 主题信息
- `themeName` / `themePath`：主题名称和路径。

## 文件清单
- `humanFiles`：用户创建的 .txt 文件，按创建时间排序。
- `aiFiles`：AI 创建的 .md 文件，按创建时间排序。

## 轮次（rounds）
每个轮次由 `humanFile` 开始、`aiFile` 结束。

- `stage`：讨论 / 执行 / 结果微调，由本地脚本根据文件结构判断。

- `humanContentPreview` / `aiContentPreview`：内容前 500 字预览，供 AI 判断分类。

## 时间与字数统计
- `totalTime` / `stageTime`：总耗时和各阶段耗时。
- `totalHumanChars` / `totalAiChars`：人输入和 AI 输出的总字符数。

---

请基于以上素材生成 `Summary.json`，详见 `summary_rules.md`。
```

#### 第二部分：内容提取与归类规则

##### 1. 单轮分类（6 类）

每轮必须且只能属于以下一类：

| 分类 | 判定标准 |
|---|---|
| 正常推进 | AI 输出符合用户预期，用户接受或仅作正向补充 |
| 细化完善 | 方向已锁定，用户回答 AI 确认性问题、填充细节 |
| 探索发散 | 用户还在思考多个方向，未收敛 |
| 纠偏拉回 | AI 漏读、误读、过度设计或实现错误，用户纠正 |
| 极端失控 | 情绪明显升级，或用户明确表达协作裂隙、要求暂停 |
| 确认空转 | 无实质信息增量，仅确认、终止或空回复 |

##### 2. 极端失控判定（第二轮内部步骤）

只在已标为「纠偏拉回」的轮次中判断。满足其一即升级为「极端失控」：
- 出现情绪化、贬损、不耐烦、剥夺协作资格的表达；
- 同一问题在多轮中无法处理，反复拉扯。

##### 3. 阶段划分（7 个标准名称）

阶段名称必须从以下集合选择：

| 阶段名称 | 判定规则 |
|---|---|
| 需求探索期 | 用户还在描述问题、提出方向，AI 提供多种方案 |
| 方案讨论期 | 围绕具体方案展开讨论，逐步收敛 |
| 方案细化期 | 方向已确定，填充实现细节 |
| 执行实施期 | 进入具体实现，讨论代码/文件/模块修改 |
| 纠偏处理期 | 连续 2 轮及以上都是纠偏/失控 |
| 结果微调期 | 功能基本完成后做细节/bug/体验微调 |
| 确认收尾期 | 主题接近尾声，用户确认收尾 |

约束：
- 每个阶段至少包含 2 个连续轮次；
- 每个轮次只属于一个阶段；
- 阶段顺序可以回退。

##### 4. 摘要规则

- `humanSummary`：1~2 句话概括用户本轮核心诉求或反馈。
- `aiSummary`：1~2 句话概括 AI 本轮核心输出。
- `overview.abstract`：用 2~4 句话概括整个主题做了什么。

##### 5. 复杂度与协作顺畅度

- `complexity`：低 / 中 / 高，基于需求范围、模块影响、边界条件判断。
- `collaboration`：顺利 / 波折 / 失控，基于纠偏轮次数量、失控轮次是否存在、整体返工量判断。
- `complexityReason` / `collaborationReason`：必须给出具体理由，不能只写结论。

##### 6. 指标计算规则

脚本已经提供总轮次、总时间、总字数。AI 只需按分类结果计数并计算比率：

```
顺畅轮次 = 正常推进 + 细化完善 + 确认空转
纠偏轮次 = 纠偏拉回 + 极端失控

byRound.intentMatchRate = 顺畅轮次 / 总轮次
byTime.intentMatchRate = 顺畅轮次耗时 / 总耗时
byChars.intentMatchRate = 顺畅轮次人输入字数 / 总人输入字数
```

##### 7. 纠偏/失控分析规则

- `perRound`：每个纠偏轮和失控轮都必须单独分析，不允许合并。
- 字段：`roundIndex`、`category`、`summary`、`rootCause`、`impact`、`lesson`。
- `summary`：1~2 句话讲清 AI 哪里错了、用户怎么纠正。
- `rootCause`：从 {漏读, 误读, 幻觉, 模型惯性, 环境差异, 未读代码} 中选一或自由描述。
- `impact`：浪费轮次、返工、方向摇摆等。
- `lesson`：必须比 summary 更抽象，能跨主题复用。
- `summary`（最终汇总）：等 perRound 全部列完后再写，指出主要模式和最关键的一条经验。

##### 8. 自检规则

生成 JSON 后，AI 必须检查：
- 所有必填字段是否存在；
- `rounds.items` 的 `roundIndex` 是否连续；
- `category` 是否在 6 类之中；
- `isExtreme` 为 `true` 的轮次，`category` 必须是「纠偏拉回」；
- `metrics` 中的比率是否在 0~1 之间；
- `analysis.correctionAnalysis.perRound` 和 `analysis.extremeAnalysis.perRound` 是否逐轮列出，没有合并。

## 五、Step 3：脚本渲染 Markdown 和 HTML

### 5.1 `Summary.md` 生成脚本

读取 `Summary.json`，按固定模板渲染为 Markdown。模板结构：

```markdown
# 经验总结：{themeName}

## 1. 总览
{abstract}
- 总耗时：{display}
- 讨论/执行/结果微调：... / ... / ...
- 复杂程度：{complexity} —— {complexityReason}
- 协作顺畅度：{collaboration} —— {collaborationReason}

## 2. 核心指标
| 维度 | 意图匹配率 | 纠偏率 | 失控率 |
|---|---|---|---|
| 按轮次 | {byRound.intentMatchRate} | {byRound.correctionRate} | {byRound.extremeRate} |
| 按时间 | {byTime.intentMatchRate} | {byTime.correctionRate} | {byTime.extremeRate} |
| 按字数 | {byChars.intentMatchRate} | {byChars.correctionRate} | {byChars.extremeRate} |

## 3. 阶段划分
| 阶段 | 轮次范围 | 主要内容 |
|---|---|---|
... ...

## 4. 轮次时间轴
| 轮次 | 类型 | 耗时 | 用户输入摘要 | AI 回复摘要 |
... ...

## 5. 纠偏分析
### R{x} —— {summary}
- 根因：...
- 影响：...
- 经验：...

## 6. 失控分析（如有）
### R{x} —— {summary}
- 触发点：...
- 升级路径：...
- 解决方式：...
- 经验：...

## 7. 经验教训
...
```

### 5.2 `Summary.html` 展示页面

一个静态 HTML 文件，读取同目录下的 `Summary.json`：

- 顶部：总览卡片 + 核心指标卡片。
- 阶段时间轴：用颜色区分阶段。
- 轮次明细表：可折叠。
- 纠偏/失控分析：高亮显示。
- 子主题耗时：条形图（如有）。

启动方式：
- 在「经验总结」窗口的详情弹窗中，「查看总结」按钮默认打开 Markdown；
- 新增一个「可视化总结」按钮，调用默认浏览器打开 `Summary.html?json=Summary.json`。
