# 测试结果分析：summary_input.json 仍有几处问题

## 测试对象

`C:\NotSystem\ClaudeCodeTool\AIProcess\需求\周报时间\.aiprocess\_tmp\summary_input.json`

## 问题一：实施确认轮次的 humanFile 匹配错误

### 现象

主轮次结构如下：

```
R0: 需求.txt -> v1.md
R1: 对v1的回复.txt -> v2.md
R2: 对v2的回复.txt -> v3.md
R3: 对v3的回复.txt -> v4.md
R4(实施确认): 对v3的回复.txt -> 实施文档.md  ← 错误
```

### 根因

当前 `Add-ImplementationRound` 取的是 `rounds.items` 中最后一个轮次的 `humanFile`。但 `rounds.items` 只包含已配对的轮次：`需求.txt→v1.md`、`对v1→v2`、……、`对v3→v4`。`对v4的回复.txt` 由于没有对应的 `v5.md`，未被纳入轮次列表。

因此实施文档前面真正的人文件（`对v4的回复.txt`）被漏掉了，程序退而求其次取了 `对v3的回复.txt`。

### 修复建议

实施确认轮次的 `humanFile` 不应该从已配对轮次里取，而应该**直接扫描目录下编号最大的 `对vN的回复.txt`**。逻辑如下：

```powershell
$lastReplyFile = Get-ChildItem -LiteralPath $Dir -File -Force |
    Where-Object { $_.Name -match '^对v(\d+)的回复\.txt$' } |
    Sort-Object { [int]($_.Name -replace '^对v(\d+)的回复\.txt$', '$1') } -Descending |
    Select-Object -First 1

if ($lastReplyFile) {
    $lastHumanFile = $lastReplyFile.Name
    $humanTime = $lastReplyFile.CreationTime  # 或从日志取
} elseif ($requirementFile) {
    $lastHumanFile = '需求.txt'
    $humanTime = $requirementFile.CreationTime
} else {
    $lastHumanFile = ''
    $humanTime = $aiTime.AddSeconds(-1)
}
```

这样即使 `对v4的回复.txt` 没有配对的 `v5.md`，实施确认轮次仍能正确指向它。

## 问题二：子单元中 `totalTime` 为 `null`

### 现象

`resultFineTunings[0].totalTime` 和 `resultFineTunings[1].totalTime` 都是 `null`。

### 根因

`Get-SubUnits` 中构造子单元对象时直接写了 `totalTime = $null`，没有调用 `Get-TotalTime` 计算。

### 修复建议

在 `Get-SubUnits` 中为每个子单元计算总时间：

```powershell
$childTotalTime = Get-TotalTime -Dir $child.FullName -Rounds $childRounds -LogEntries $LogEntries -ThresholdMinutes $thresholdMinutes

$subUnits += [PSCustomObject]@{
    ...
    totalTime = $childTotalTime
    ...
}
```

注意：`thresholdMinutes` 需要传入 `Get-SubUnits`。

## 问题三：`stageTime.execution` 和 `stageTime.tweak` 长期为 0

### 现象

```json
"stageTime": {
  "discussion": { "minutes": 45, ... },
  "execution": { "minutes": 0, ... },
  "tweak": { "minutes": 0, ... }
}
```

### 根因

当前 `Get-StageTime` 只计算了 discussion 时长，execution 和 tweak 被硬编码为 0。这是旧逻辑遗留，没有利用新的 `resultFineTunings` 和 `subThemes` 结构。

### 修复建议

按语义拆分三阶段：

| 阶段 | 来源 |
|---|---|
| `discussion` | 根目录 `rounds.items` 的活跃时长（不含实施确认） |
| `tweak` | `resultFineTunings` 所有子单元的活跃时长之和 |
| `execution` | `subThemes` 所有子单元的活跃时长之和 |

实现上，可以复用 `Get-TotalTime` 分别计算根目录轮次、结果微调轮次、子主题轮次的时长，然后汇总到 `stageTime`。

## 问题四：`files` 列表只包含根目录文件

### 现象

```json
"files": {
  "humanFiles": [需求.txt, 对v1的回复.txt, ..., 对v4的回复.txt],
  "aiFiles": [v1.md, v2.md, v3.md, v4.md]
}
```

这个列表没有包含 `结果微调/01/`、`结果微调/02/` 里的文件，也没有包含 `实施步骤/` 下的文件。

### 分析

当前 `files` 字段的设计意图不够明确：

- 如果它是「主题下所有相关文件清单」，那应该递归包含子目录；
- 如果它只是「根目录沟通主线文件清单」，那它的价值有限，因为 `rounds.items` 已经列出了这些文件。

### 修复建议

有两个方向，需要确认：

**方向 A：删除或弱化 `files` 字段**

既然 `rounds.items`、`resultFineTunings`、`subThemes` 已经详细列出了每轮对应的文件，`files` 字段的意义不大。可以考虑：

- 删除 `files` 字段；或
- 仅保留 `humanFiles` / `aiFiles` 作为根目录文件清单，并在 `summary_input_guide.md` 中明确说明「完整文件请按各轮次路径读取」。

**方向 B：扩展 `files` 字段为全量文件树**

如果希望给 AI 一个全局文件视图，可以改成递归结构：

```json
"files": {
  "humanFiles": [...],
  "aiFiles": [...],
  "resultFineTuningFiles": {
    "01": { "humanFiles": [...], "aiFiles": [...] },
    "02": { "humanFiles": [...], "aiFiles": [...] }
  },
  "subThemeFiles": {
    "01-xxx": { "humanFiles": [...], "aiFiles": [...] }
  }
}
```

但这会增加数据冗余，因为子单元的文件已经在 `rounds.items` 中体现。

我的倾向是**方向 A**：

- 保留 `files` 字段但说明它只是根目录文件清单；
- 或者干脆删除，避免给 AI 造成「这就是全部文件」的误导。
