# 回复确认：实施确认归入主轮次，子主题即实施步骤

## 一、理解确认

1. **子主题一般就在「实施步骤/」下面**，所以 `实施步骤/XX/` 就是 `subThemes`。
2. **`实施确认` 不需要单独独立出来**，它本来就是正常轮次的一部分，应该放进 `rounds.items`。
3. **`实施文档.md` 不仅存在于根目录**，在子主题（实施步骤/XX/）和结果微调（结果微调/XX/）下面也都可能存在。
4. 子主题和结果微调的结构与主体结构一致。

## 三、回答你的四个问题

### (a) 你认为应该包含哪些东西？

`summary_input.json` 应该包含：

- 主题根目录下的全部轮次（含「实施确认」）；
- `resultFineTunings`：每个结果微调子目录的完整轮次（含其实施确认）；
- `subThemes`：每个实施步骤子目录的完整轮次（含其实施确认）。

每个子单元的结构和主体结构一致，都有 `rounds.items` 和 `rounds.phases`。

### (b) 子主题和结果微调的讨论轮次算不算时间、算不算时长？

**算。**

理由：

- 这些讨论都是围绕同一个大主题展开的，属于该主题的总工作量；
- 如果只算根目录轮次，会把一个可能耗时几小时的复杂主题统计成十几分钟，严重失真；
- 可以在 `overview.totalTime` 里放总时长，同时在 `stageTime` 或额外字段里拆分：
  - 主讨论时长
  - 结果微调总时长
  - 子主题总时长

### (c) 意图匹配率、纠偏率、失控率这些指标，子主题和结果微调算吗？

**算。**

理由：

- 子主题和结果微调里的轮次同样是人与 AI 的协作过程；
- 如果某个子主题里出现了大量纠偏或极端失控，而主主题指标里被排除，那主主题看起来就会很「顺利」，掩盖真实问题；
- 指标计算应该以「该主题下所有有效轮次」为分母，包括根目录、结果微调、子主题。

### (d) 把它们剔除掉，最终结论会准确吗？

**不会准确。**

如果剔除：

- 总时长会被人为压缩；
- 复杂度、协作顺畅度会被高估；
- 问题模式和经验教训会丢失；
- 最终总结会变成一个只反映表层讨论的「虚假顺利」报告。

## 四、指标计算方式建议

为了既保留总体指标，又能看清各部分的贡献，建议这样设计：

### `metrics` 主层级

```json
"metrics": {
  "totalRounds": 12,
  "totalTimeMinutes": 180,
  "totalHumanChars": 5000,
  "totalAiChars": 15000,
  "byRound": { "normal": 8, "refinement": 2, ... },
  ...
}
```

这里的 `totalRounds` 和分类统计是**全量合并**后的结果。

### 可选：各部分拆分

```json
"breakdown": {
  "main": { "rounds": 5, "timeMinutes": 60 },
  "resultFineTunings": { "rounds": 4, "timeMinutes": 60 },
  "subThemes": { "rounds": 3, "timeMinutes": 60 }
}
```

这个拆分字段可选，但如果要做完整总结，建议加上。
