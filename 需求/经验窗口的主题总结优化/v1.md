# 关于「经验窗口的主题总结优化」的理解与方案 v1

## 一、当前逻辑流程（基于代码梳理）

我先把项目整体过了一遍，重点看了「经验总结」窗口和主题总结生成链路。当前流程可以概括为：

1. **入口**：托盘菜单 → `经验总结` → 打开 `SummaryWindow.ahk` 的 ListView 窗口。
2. **触发**：选中主题后点击「生成总结」，进入 `SummaryGenerator.ahk` 的 `GenerateSummary()`。
3. **数据收集**：调用 `PrepareSummaryData.ps1`，扫描主题目录，输出如下元数据：
   - 文件总数、字符总量、最后修改文件/时间
   - 总耗时、讨论/执行/结果微调三阶段耗时
   - 文件树 `fileTree`、核心文件清单 `coreFiles`、嵌套子主题 `subThemes`
   - 操作日志路径（`log.jsonl`）
4. **Prompt 拼装**：用 `summary_prompt.txt` + `summary_template.md` 生成完整提示词。
5. **Agent 生成**：通过 `AgentDispatcherSend()` 把 Prompt 发给绑定的 `SummaryAgent`，由 Agent 直接输出 `Summary.md`。
6. **完成通知**：Agent 写完后执行 `NotifySummaryComplete.ps1`，刷新窗口。

也就是说，**现在的总结本质上是「数据收集 + 让 Agent 自由写 Markdown」**。分类、摘要、问题提炼、经验教训全部依赖大模型一次性完成。

## 二、对需求文档和 DeepSeek 讨论要点的理解

你的核心诉求可以归纳为三点：

1. **结构化分析轮次**：把一次主题里的「需求 → AI 回复 → 人再回复」识别为一个个轮次，并做时间轴呈现。
2. **轮次分类 + 量化指标**：区分正常、错位、极端失控、细化、重复确认等类型；并计算意图匹配率、错误轮次数量、失控轮次数量等。
3. **输出形式升级**：最终输出一个结构化的 JSON 文件，再由 HTML 页面渲染展示，而不是让 Agent 直接写 Markdown。

DeepSeek 文档里还提到另外一套分类（试探、突破、细化、空转、纠偏），我觉得和你列的五类不完全冲突，可以统一到同一层分类体系里。

## 三、我的想法：把重活从 Agent 移到本地脚本，Agent 只做「判断」

现在每个总结都要消耗 20~40 个 token，一大原因是 Agent 要把目录下所有文件读一遍、自己抽象轮次、再写长篇 Markdown。要降低 token，关键是在本地 AHK/PowerShell 层先把「轮次」和「元数据」抽好，Agent 只负责做一件它擅长但脚本做不到的事：**阅读每轮内容并判断这轮属于哪一类**。

### 3.1 轮次提取（本地完成）

项目里其实已经有轮次的约定，只是没有显式对象。一轮可以定义为：

- 人的输入：`需求.txt`（第 0 轮）或 `对vN的回复.txt`
- AI 的输出：`v(N+1).md`

本地脚本可以按现有 `SortByConvention` 的顺序把文件配成一对一对，形成轮次列表：

```
Round 0: 需求.txt → v1.md
Round 1: 对v1的回复.txt → v2.md
Round 2: 对v2的回复.txt → v3.md
...
Round N-1: 对v(N-1)的回复.txt → vN.md
```

每个轮次附上：

- 轮次编号
- 阶段归属（讨论 / 执行 / 结果微调，可直接复用 `PrepareSummaryData.ps1` 的阶段边界）
- 人输入文件路径、AI 输出文件路径
- 起止时间（从文件创建/修改时间或 `log.jsonl` 中提取）
- 字数/字符数

### 3.2 轮次分类（Agent 轻量判断）

脚本把每轮的内容（人的输入 + AI 的输出，按固定长度截断，避免太长）拼成一个「待分类列表」Prompt，让 Agent 输出一个 JSON 数组：

```json
[
  {
    "roundIndex": 0,
    "category": "正常",
    "deviationRootCause": null,
    "summary": "用户提出需求，AI 理解正确并给出实施文档。"
  },
  {
    "roundIndex": 3,
    "category": "纠偏/控制",
    "deviationRootCause": "漏读",
    "summary": "AI 忽略了配置对话框的已有约定，需要用户拉回。"
  }
]
```

分类标准建议先统一为下面这一套，避免太碎：

| 大类 | 含义 |
|---|---|
| 正常 | AI 一次理解，无需纠正 |
| 细化 | 方向已明确，继续打磨细节 |
| 试探 | 探索多个方向，尚未收敛 |
| 纠偏/控制 | AI 理解有偏差，用户需要拉回 |
| 极端失控 | 反复纠正无效、出现情绪化指令、协作裂隙 |
| 空转/重复确认 | 没有信息增量，只是确认或重复 |

> 如果你更偏好「试探 / 突破 / 细化 / 空转 / 纠偏」这套说法，我可以把分类名称按这个来，内涵上基本能覆盖。

### 3.3 量化指标（本地计算）

分类结果返回后，本地脚本直接计算：

- 总轮次数
- 正常轮次数 / 占比（意图匹配率）
- 纠偏轮次数 / 占比
- 极端失控轮次数
- 空转轮次数 / 占比
- 每轮平均耗时、各阶段耗时占比（可选）

### 3.4 最终输出：结构化 JSON

我的建议是，最终产物从现在的 `Summary.md` 改成 `{themePath}\.aiprocess\Summary.json`，结构类似：

```json
{
  "themeName": "历史财富",
  "themePath": "C:\\...\\历史财富",
  "overview": {
    "abstract": "...",
    "complexity": "高",
    "smoothness": "波折",
    "tags": ["..."],
    "totalTime": "0天 9小时 24分钟"
  },
  "metrics": {
    "totalRounds": 42,
    "normalRounds": 28,
    "intentMatchRate": 0.67,
    "deviationRounds": 10,
    "extremeRounds": 1,
    "idleRounds": 3
  },
  "timeline": [
    {
      "roundIndex": 0,
      "stage": "讨论",
      "humanFile": "需求.txt",
      "aiFile": "v1.md",
      "startTime": "2026-06-27 14:14:05",
      "endTime": "2026-06-27 14:28:41",
      "category": "正常",
      "summary": "..."
    }
  ],
  "patterns": {
    "mainIssue": "早期对项目结构理解不足导致多轮低质量确认",
    "keyInsight": "大需求启动前应先梳理项目代码和现有约定"
  },
  "issues": [...],
  "lessons": [...]
}
```

其中 `overview.abstract`、`patterns`、`issues`、`lessons` 仍然可以由 Agent 生成，但 Agent 不再是「读所有文件自由发挥」，而是基于本地已经抽好的轮次分类和时间轴来写，输入长度和不确定性都会下降。

### 3.5 HTML 展示

另外写一个静态 HTML 文件（可以放在 `app/templates/summary/SummaryViewer.html`），读取 `Summary.json`，用时间轴、进度条、饼图/柱状图把轮次和指标可视化。展示内容可以包括：

- 顶部总览卡片（耗时、轮次、匹配率、失控数）
- 轮次时间轴（用不同颜色区分分类）
- 阶段耗时分布
- 轮次分类占比
- 关键经验教训列表

### 3.6 Token 为什么能减少

| 现在 | 优化后 |
|---|---|
| Agent 读全量文件、自己理解结构、自己写完整 Markdown | 本地脚本读文件并抽轮次，Agent 只读「每轮摘要 + 分类结果」 |
| 输出不受控，容易产生无结构内容 | 输出固定 JSON Schema，HTML 负责呈现 |
| 每次总结都重新理解一次项目 | 本地层把项目约定编码进脚本，Agent 专注于内容判断 |

我估计优化后单主题的 Agent 输入可以下降 30%~60%，具体取决于主题大小。

## 四、需要和你确认的问题

下面这些问题，有些我已经能从代码里确认，有些需要你拍板：

### 4.1 轮次定义

- 一轮是否严格等于「人的输入文件 → AI 的 vN.md」？
- `实施文档.md` 本身是否算一轮，还是算作「讨论阶段结束的标志」？
- `结果微调/` 下的子目录，是否单独提取成一套新的轮次序列？（我倾向于是，因为它是独立的循环。）

### 4.2 分类标准

- 你更想沿用你列的「正常 / 控制 / 极端失控 / 深根细化 / 重复确认」五类，
  还是 DeepSeek 的「试探 / 突破 / 细化 / 空转 / 纠偏」五类，
  还是我把两者合并成一套？
- 「控制轮次」和「极端失控轮次」的边界，你是希望由 Agent 判断，还是你给出更明确的判定规则？

### 4.3 时间来源

- 轮次时间优先用文件创建/修改时间，还是从 `log.jsonl` 里找对应操作记录？
- `log.jsonl` 目前记录了 `建需求`、`建回复`、`复执行`、`完成通知` 等动作，但不一定能精确对应到每一轮。如果文件时间和日志时间冲突，以哪个为准？

### 4.4 输出与兼容

- 生成 `Summary.json` 后，是否还要保留一份人类可读的 `Summary.md`？还是完全用 HTML 替代？
- HTML 查看入口放在哪里？是经验总结窗口里加一个「查看可视化总结」按钮，还是直接替换现有「查看总结」按钮的行为？

### 4.5 实施范围

- 是先在当前主题上做一个可运行的原型验证，验证通过后再推广到所有主题？
- 还是希望直接修改正式的数据收集和总结生成链路？

## 五、下一步建议

如果你认可这个方向，我建议第一步先做：

1. 在本主题目录下写一个独立的 PowerShell/AHK 原型脚本，只负责把「历史财富」这个主题的轮次提取出来、按你认可的分类标准输出一个临时 JSON。
2. 我们人工校验这个 JSON 的分类是否符合你的直觉。
3. 分类标准定下来后，再写 HTML 渲染层和正式的生成链路。

这样可以用最小的改动先验证「轮次能不能被正确识别、分类标得准不准」，避免一上来就改正式代码。
