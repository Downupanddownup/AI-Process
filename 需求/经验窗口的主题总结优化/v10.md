# 修正 phases 分组逻辑 + 为极端失控轮补充判定原因 v10

收到，两个核心问题我重新调整。

## 一、关于 phases 的重新理解

你说得对，我之前的 `phases` 太拘泥于时间顺序，而且 `mainCategories` 把「极端失控」和「正常推进」放在一起确实没有意义。

我重新理解：`phases` 不是时间轴切片，而是**按沟通主题/目的对轮次做聚合**。如果前面和后面的轮次是同一类事情，就可以合并在一起，不一定要连续。

因此我提议：

1. **去掉 `mainCategories` 字段**：它容易混淆，价值不高。
2. **`roundRange` 改为 `roundIndexes` 数组**：支持非连续轮次归入同一阶段。
3. **阶段按「主题目的」命名**：比如「需求探索」「方案细化」「核心能力验证」，而不是按时间顺序硬切。

### 以「闭环」为例重新分组

```json
"phases": [
  {
    "name": "需求探索",
    "roundIndexes": [0],
    "summary": "用户提出让 AI Process 形成闭环的想法：AI 生成 Markdown 后自动用 IDEA 打开。"
  },
  {
    "name": "方案细化",
    "roundIndexes": [1, 2, 5, 6],
    "summary": "明确触发对象、复选框形式、与现有 IDEA 开关关系、默认状态、脚本位置、提示词填充方式等实现细节。"
  },
  {
    "name": "核心能力验证",
    "roundIndexes": [3, 4],
    "summary": "用户对 AI 直接执行命令的可靠性产生质疑，要求先验证核心能力；验证可行后改用 PowerShell 脚本封装。"
  }
]
```

这样分组的价值在于：
- 一眼看出这个主题经历了几次「主题切换」。
- 「核心能力验证」作为一个独立阶段被凸显出来，说明它是一次关键的协作危机。
- 不再是「第几轮到第几轮在干什么」这种流水账。

## 二、为极端失控轮补充判定原因

你建议在每个极端失控轮次里补一个判定原因。我提议在 `rounds.items` 中增加 `extremeReason` 字段：

```json
{
  "roundIndex": 3,
  "stage": "讨论",
  "humanFile": "对v3的回复.txt",
  "aiFile": "v4.md",
  "startTime": "2026-06-17 00:58:02",
  "endTime": "2026-06-17 01:00:21",
  "durationMinutes": 2,
  "humanChars": 469,
  "aiChars": 4143,
  "humanSummary": "...",
  "aiSummary": "...",
  "category": "纠偏拉回",
  "isExtreme": true,
  "extremeReason": "用户宣布协作前提：如果 AI 不能稳定打开 Markdown，后续所有沟通无效；属于情绪化协作裂隙声明。"
}
```

约束：
- 只有 `isExtreme` 为 `true` 的轮次才需要填写 `extremeReason`。
- 内容必须简明扼要，1~2 句话说明判定依据。
- 判定依据必须从两个标准里选一或同时说明：
  1. 出现情绪化、贬损、不耐烦、剥夺协作资格等表达；
  2. 同一问题在多轮中无法处理、反复拉扯。
