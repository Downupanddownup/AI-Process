# 执行结果微调 - 实施文档

## 1. 背景与目标

当前 `AI Process` 已经具备完整的需求沟通与执行框架：

1. `需求.txt`
2. `vX.md`
3. `对vX的回复.txt`
4. `实施文档.md`
5. `复执行`

这套框架本身已经足够处理：

1. 问题现状确认
2. 影响范围分析
3. 原因讨论
4. 方案收敛
5. 最终实施

因此，当正式执行后出现局部问题、边界瑕疵或结果微调需求时，本次不新增一套独立流程，也不新增“分析引擎式”能力，而是：

`复用现有流程，只补一个从主题目录快速进入“实施结果问题子目录”的轻量目录切换机制。`

目标是让用户在不打断现有工作习惯的前提下，可以把执行结果中的每一个局部问题，作为一个独立的小问题子目录重新走一遍当前成熟流程。

## 2. 需求边界

### 2.1 本次要解决的问题

本次只解决“执行结果微调”的目录组织和轻量切换能力，具体包括：

1. 在主题目录下快速创建结果微调问题子目录。
2. 自动切换到新建的问题子目录。
3. 在问题子目录下继续复用现有：
   - `建需求`
   - `复需求`
   - `建回复`
   - `复回复`
   - `复关系`
   - `复执行`
4. 在问题子目录处理完成后，快速返回主题目录。
5. 在界面上用极简状态标记提示当前处于问题子目录。

### 2.2 本次不解决的问题

本次不处理以下内容：

1. 不新增新的分析模板体系。
2. 不自动创建 `需求.txt`。
3. 不自动生成 `vX.md` 或 `实施文档.md`。
4. 不新增复杂的“结果微调流程模式配置”。
5. 不让用户停留在“结果微调总目录”这一中间层。
6. 不改变现有正式执行流程逻辑。

### 2.3 必须保持不变的约束

1. 工具仍然保持轻量，不做重型流程编排。
2. 现有文件级操作按钮尽量复用，不新增重复按钮。
3. 结果微调能力本质是目录级入口，不是新流程系统。
4. UI 必须继续服从当前紧凑小面板的布局约束。

## 3. 目录模型

### 3.1 磁盘结构模型

在主题目录下，结果微调相关目录结构固定为：

```text
主题目录/
  结果微调/
    01/
    02/
    03/
```

说明：

1. `结果微调` 是固定总目录名。
2. 每个子目录代表一个独立结果问题。
3. 子目录名使用两位纯数字编号：
   - `01`
   - `02`
   - `03`

### 3.2 工具交互模型

虽然磁盘上存在三层结构，但工具层只认两种状态：

1. `主题目录状态`
2. `结果微调问题子目录状态`

`结果微调` 总目录本身只是存储层，不是用户交互层，不应成为用户停留目录，也不应成为工具显式工作态。

## 4. 状态定义与判定

### 4.1 主题目录状态

当当前目录不是结果微调问题子目录时，工具默认认为当前处于主题目录状态。

在这个状态下：

1. 显示 `设目录`
2. 显示 `建问题`
3. 隐藏 `返`
4. 不显示子目录状态标记

### 4.2 问题子目录状态

当当前目录满足以下结构规则时，认定为结果微调问题子目录：

1. 当前目录名为两位纯数字，如 `01`、`02`、`03`
2. 当前目录的父目录名为固定字符串 `结果微调`

只有同时满足这两个条件，才算问题子目录状态。

这样可以避免把普通项目中的其他数字目录误判为结果微调目录。

### 4.3 路径结构判定原则

状态判定必须使用：

`当前路径结构判定`

而不能依赖：

1. 是否由程序刚创建
2. 是否由用户通过某个按钮进入
3. 程序内存里记住的上次状态

原因：

1. 用户可能手动切目录。
2. 程序可能重启。
3. 当前目录可能来自外部已有结构。

只要路径结构满足约定，就应认它是问题子目录。

### 4.4 子目录状态下的目录切换约束

在问题子目录状态下：

1. 不再保留 `设目录` 入口。
2. 用户如果想切到其他目录，必须先点击 `返` 回到主题目录。
3. 回到主题目录后，才允许继续：
   - `设目录`
   - `建问题`
   - 或其他主目录级动作

## 5. 核心交互方案

### 5.1 `建问题` 按钮

本次新增或替换后的按钮名称固定为：

`建问题`

其真实语义不是“建任意目录”，而是：

`在当前主题目录下，创建并进入下一个结果微调问题子目录。`

### 5.2 `建问题` 按钮行为

仅在主题目录状态下可见或可用。

点击后执行以下逻辑：

1. 检查当前目录是否已设置。
2. 以当前目录为主题目录。
3. 检查主题目录下是否存在 `结果微调` 总目录。
4. 如果不存在，则自动创建 `结果微调`。
5. 在 `结果微调` 中查找已有的两位编号问题子目录。
6. 计算下一个可用编号：
   - 若无子目录，则创建 `01`
   - 若已有 `01`、`02`，则创建 `03`
7. 创建该问题子目录。
8. 自动将当前目录切换到该新子目录。
9. 刷新当前目录显示与界面状态。

### 5.3 `返` 按钮

本次新增一个极短返回按钮：

`返`

其语义固定为：

`从当前结果微调问题子目录，直接返回主题目录。`

### 5.4 `返` 按钮行为

仅在问题子目录状态下可见或可用。

点击后执行以下逻辑：

1. 读取当前目录。
2. 确认它满足问题子目录结构。
3. 取当前目录的父目录的父目录，作为主题目录。
4. 将当前目录切换回该主题目录。
5. 刷新当前目录显示与界面状态。

### 5.5 工具层不允许停留在 `结果微调` 总目录

这点必须明确：

1. `建问题` 不会把当前目录切到 `结果微调`
2. `返` 不会把当前目录切到 `结果微调`
3. 工具交互只在主题目录和问题子目录之间来回切换
4. 子目录状态下若需切到其他目录，必须先 `返`

## 6. 界面方案

### 6.1 新增按钮位置原则

由于当前主面板空间非常紧，新增能力必须尽量小而准，不允许引入冗余说明。

因此：

1. 只增加必要控件。
2. 按钮文案尽量短。
3. 状态标记尽量短。
4. 不新增多行说明文字。

### 6.2 显示规则

#### 主题目录状态

1. 显示 `设目录`
2. 显示 `建问题`
3. 隐藏 `返`
4. 隐藏子目录状态标记

#### 问题子目录状态

1. 隐藏 `设目录`
2. 隐藏 `建问题`
3. 显示 `返`
4. 显示子目录状态标记

### 6.3 当前目录状态标记

状态标记只在问题子目录状态下显示。

默认推荐符号：

`↳`

原因：

1. 很短。
2. 很醒目。
3. 语义上最贴近“主目录派生出的子问题目录”。
4. 比 emoji 更稳，更符合当前工具风格。

如果实施时在当前控件字体下显示不理想，可退到：

1. `◆`
2. `▣`
3. `子`

但默认方案应先尝试 `↳`。

### 6.4 图标/emoji 方案边界

本次不默认采用 emoji 方案。

原因：

1. 经典 AHK GUI 控件里 emoji 显示稳定性不如普通 Unicode 符号。
2. 工具整体风格偏朴素效率型，不适合引入过强装饰性。
3. 状态标记本质只是轻量提醒，不值得为此引入更复杂的资源或兼容处理。

## 7. 与现有能力的关系

### 7.1 保留并复用的按钮

在问题子目录中，以下按钮和流程继续按现有逻辑工作：

1. `建需求`
2. `复需求`
3. `建回复`
4. `复回复`
5. `复关系`
6. `复执行`

也就是说：

问题子目录只是一个新的“主题目录替代工作位”，一旦切进去，后续文件流程完全复用现有成熟能力。

### 7.2 不应自动做的事情

以下事情本次不自动做：

1. 不自动创建 `需求.txt`
2. 不自动复制需求提示词
3. 不自动帮用户开讨论
4. 不自动帮用户生成 `实施文档.md`

原因是这些能力已有现成按钮，继续复用最轻最稳。

## 8. 可落地技术方案

### 8.1 需要修改的主要文件

预计正式实施时主要修改：

1. `C:\NotSystem\ClaudeCodeTool\AIProcess\app\AIProcess.ahk`

本次不一定需要新增模板文件，因为这次新增的是目录级能力，不是提示词级新流程。

### 8.2 主脚本新增能力

主脚本中需要补充以下能力：

1. 结果微调目录常量
2. 结果问题子目录识别函数
3. 主题目录计算函数
4. 下一个编号子目录计算函数
5. 创建并进入问题子目录函数
6. 返回主题目录函数
7. UI 状态刷新函数

### 8.3 推荐新增的常量/辅助函数

建议引入稳定常量，例如：

1. 结果微调总目录名：
   - `结果微调`

建议新增辅助函数，例如：

1. `IsResultIssueDir(dirPath)`
   判断给定路径是否为结果微调问题子目录。

2. `GetThemeRootFromIssueDir(dirPath)`
   从问题子目录反推主题目录。

3. `GetResultIssueRoot(dirPath)`
   在主题目录下获取 `结果微调` 总目录路径。

4. `GetNextIssueDirName(issueRootPath)`
   计算下一个两位编号目录名。

5. `CreateAndEnterIssueDir()`
   创建并切入新问题子目录。

6. `ReturnToThemeDir()`
   从当前问题子目录返回主题目录。

7. `RefreshDirectoryStateUI()`
   根据当前目录状态更新：
   - 目录文本
   - 状态标记
   - `设目录`
   - `建问题`
   - `返`

### 8.4 子目录判定实现建议

`IsResultIssueDir(dirPath)` 建议按以下顺序判断：

1. 路径必须存在且是目录。
2. 取当前目录名。
3. 取父目录名。
4. 当前目录名需匹配正则：
   - `^\d{2}$`
5. 父目录名必须等于：
   - `结果微调`

满足则返回真，否则返回假。

### 8.5 下一个编号计算建议

`GetNextIssueDirName(issueRootPath)` 应：

1. 枚举 `结果微调` 目录下的直接子目录。
2. 只识别两位数字目录。
3. 找到最大编号。
4. 返回下一个编号，补零为两位。

例如：

1. 无目录 -> `01`
2. 已有 `01` -> `02`
3. 已有 `01`、`02`、`03` -> `04`

### 8.6 返回主题目录建议

当当前目录是：

```text
主题目录\结果微调\01
```

返回主题目录的算法应直接取：

1. 当前目录的父目录 = `结果微调`
2. 再上一级目录 = 主题目录

并将 `CurrentDir` 设置为该主题目录。

### 8.7 当前目录显示更新建议

当前已有 `UpdateCurrentPathDisplay()`，它只显示：

1. `当前：未设置`
2. `当前：目录名`

本次建议在此基础上扩展为：

1. 主目录态：
   - `当前：目录名`

2. 问题子目录态：
   - `当前：↳目录名`
   或其他等价的极简表达

注意：

1. 这里只做极简状态提示。
2. 不要把整段说明塞进目录文本。

### 8.8 控件显示/隐藏策略实现建议

实施时可以有两种方式：

1. `Visible` 切换
2. 同位置控件互斥显示

由于你已经明确偏好“隐藏掉无意义按钮”，本次建议直接按状态切换显隐，而不是长期显示后禁用。

推荐规则：

1. 主目录态：
   - `建问题` Visible = true
   - `返` Visible = false

2. 问题子目录态：
   - `建问题` Visible = false
   - `返` Visible = true

## 10. 验收标准

### 10.1 目录创建与切换验收

1. 在主题目录状态下，可使用 `建问题`
2. 首次点击时，若不存在 `结果微调` 目录，会自动创建
3. 自动创建 `01` 子目录并切换进去
4. 再次回到主题目录后继续点击，可创建 `02`、`03`

### 10.2 状态判定验收

1. 普通主题目录下，不显示子目录标记
2. 进入 `结果微调\01` 后，正确识别为子目录状态
3. 进入其他普通数字目录，不应误判为结果微调问题子目录

### 10.3 UI 验收

1. 主目录态显示 `设目录`
2. 主目录态显示 `建问题`
3. 主目录态隐藏 `返`
4. 子目录态隐藏 `设目录`
5. 子目录态隐藏 `建问题`
6. 子目录态显示 `返`
7. 子目录态显示 `↳` 状态标记

### 10.4 流程复用验收

1. 进入问题子目录后，`建需求` 仍然正常创建该子目录下的 `需求.txt`
2. `复需求`、`建回复`、`复回复`、`复关系`、`复执行` 均继续工作
3. `返` 返回主题目录后，原主流程目录继续可正常使用

### 10.5 边界验收

1. 未设置当前目录时，不应允许误触目录级动作
2. 在子目录状态下，不应还能直接显示 `建问题`
3. 在主目录状态下，不应显示无意义的 `返`
4. 在子目录状态下，不应继续显示 `设目录`
5. 如需切换到其他目录，必须先 `返` 回主题目录

## 11. 实施顺序建议

正式进入代码实施阶段时，建议按下面顺序执行：

1. 先补路径结构判定函数。
2. 再补“结果微调目录 / 下一个编号目录”计算逻辑。
3. 再实现 `建问题` 和 `返` 的目录切换行为。
4. 再补状态标记和按钮显隐逻辑。
5. 最后做主目录/子目录来回切换回归验证。

## 12. 结论

本次需求的本质，不是新增一个“执行后分析系统”，而是：

`在不改变现有文件沟通流程的前提下，为执行结果中的局部问题，增加一个从主题目录快速进入独立问题子目录、并能快速返回的轻量目录级闭环。`

该闭环的关键在于：

1. 文件流程完全复用现有能力
2. 工具层只认主目录和问题子目录两种状态
3. `结果微调` 总目录只作为存储结构，不作为交互层
4. 界面新增能力必须极简、稳定、易懂

只要按本文档实施，这个“执行结果微调”能力就能以最轻的方式融入当前工具，而不会把整个项目做重。
