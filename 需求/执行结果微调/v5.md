## 1. 关于“建目录”按钮的命名

你说得对，`建目录` 这个名字太泛了，不够直白。

因为这个按钮实际上不是“建任意目录”，而是：

`围绕实施结果，创建一个新的问题处理子目录并切进去。`

所以它的名字应该尽量表达“新开一个问题处理入口”，而不是表达“文件系统动作”。

### 我当前更倾向的候选

如果限定尽量短、尽量适合当前紧凑 UI，我会优先考虑下面几组：

1. `建问题`
2. `开问题`
3. `新问题`
4. `建子题`

### 我的排序判断

我目前最推荐的是：

1. `建问题`

原因：

1. 足够短。
2. 含义直白。
3. 更符合你真实动作，不会让人以为只是文件系统建目录。
4. 后续回到主目录后，再点一次 `建问题` 去开 `02`、`03`，心智也很顺。

我不太推荐：

1. `建目录`
   太偏技术动作。

2. `建子题`
   有点书面，而且“子题”这个词不如“问题”自然。

3. `新问题`
   也能用，但更像状态说明，不像动作按钮。

4. `开问题`
   比较口语，力度够，但略微不如“建问题”稳。

所以如果你让我现在先定一版，我会先押：

`建问题`

## 2. 关于按钮显示/隐藏策略

你这轮已经把这点定得比较清楚了，我这里帮你收成规则。

### `建问题` 按钮

1. 只在主题目录状态下显示或可用。
2. 在问题子目录状态下应隐藏。

我更偏向“隐藏”而不是“禁用”，因为：

1. 当前 UI 空间紧。
2. 子目录里这个按钮没有意义。
3. 隐藏后界面语义更干净。

### `返` 按钮

1. 只在问题子目录状态下显示。
2. 在主题目录状态下隐藏。

这个判断我认同，因为：

1. 主目录里它没有意义。
2. 你已经明确想把界面尽量精简。
3. `返` 本身就已经足够短，配合只在子目录中出现，语义会很清楚。

所以这里我现在更倾向：

1. 主目录态：显示 `建问题`，不显示 `返`
2. 子目录态：显示 `返`，不显示 `建问题`

这会比“两者都在，只是禁用一个”更利落。

## 3. 关于子目录状态判定

你提的思路我认为是对的，而且应该就是这么做。

也就是：

`直接读取当前目录路径，再判断它是否满足结果微调问题子目录的结构规则。`

我认为这里不应该依赖额外配置、额外标记文件或内存状态。

### 我更推荐的判定原则

应当走“路径结构判定”，而不是“历史操作判定”。

也就是说，不要用：

1. “是不是刚才通过按钮进去的”
2. “是不是程序自己创建的”

而应该用：

1. 当前路径是否符合约定结构

这样更稳，因为：

1. 用户可能手动切路径。
2. 程序可能重启。
3. 当前目录可能来自外部已有目录。

如果只要结构满足，就认它是问题子目录，工具就会更健壮。

### 我建议的判定标准

如果目录结构约定为：

```text
主题目录/
  结果微调/
    01/
```

那么当前目录要被认定为“问题子目录”，至少应满足：

1. 当前目录名是两位纯数字，如 `01`、`02`、`03`
2. 当前目录的父目录名是固定的结果微调目录名，例如 `结果微调`

也就是同时满足这两个条件，才算子目录状态。

这样比“只看当前目录名是不是两位数字”更安全。

否则普通项目里也可能有别的 `01` 目录，会误判。

### 反向得到主题目录

一旦确认当前目录是：

```text
主题目录/结果微调/01
```

那么主题目录就可以稳定地取：

`当前目录的父目录的父目录`

这也正好支持 `返` 按钮直接回主题目录。

## 4. 关于状态标记：图标还是文字

我认为：

`可行，而且技术上不难，但应优先选最稳的实现方式。`

### 先说可行性

对于当前这个 AHK 小工具来说，显示一个简单图标或符号，技术上是可做的，不属于困难功能。

因为本质上它只是一个很小的 UI 标记，可以通过：

1. 文本字符
2. 特殊符号
3. 很短的标签

来实现。

如果你说的“图标”是指：

1. 一个小符号
2. 一个醒目的 Unicode 图形字符

那这个成本很低。

### 但我不建议一上来做复杂图标方案

如果“图标”指的是：

1. 真正的图片资源
2. 图标文件
3. 额外加载 UI 资源

那我当前不建议。

原因很简单：

1. 这会让实现复杂度上去。
2. 当前工具整体风格很朴素。
3. 这个状态提示本身只是辅助，不值得为了它引入重资源。

### 我更推荐的做法

优先用：

1. 极短文字
2. 或简单符号

例如：

1. `子`
2. `微`
3. `*`
4. `•`

我目前更偏向：

1. 先用一个非常短的文字或符号实现
2. 只要它足够醒目即可
3. 后面如果觉得不够，再升级成更像“图标”的表达

这样更稳。

## 5. 我当前更倾向的一版界面行为

如果现在要我给一个比较完整、但仍然克制的界面判断，我会这样定义：

### 主题目录状态

1. 正常显示当前目录名称
2. 显示 `建问题`
3. 隐藏 `返`
4. 不显示子目录标记

### 问题子目录状态

1. 正常显示当前目录名称
2. 隐藏 `建问题`
3. 显示 `返`
4. 显示一个极简子目录标记

这样界面会自然表达“当前你在哪一层”，而且不会新增很多说明文字。

## 6. 我现在还剩下的一个关键问题

我觉得这轮之后，大部分东西都快收敛了，现在只剩最后一个比较值得钉死的点：

### 结果微调总目录的名字，到底叫什么

因为前面虽然一直说“结果微调”，但这个名字本身你好像还没有最终钉死。

既然状态判定要依赖父目录名，后续实现也要依赖它，那这个名字最好现在就定。

我建议你从这几个里选一个：

1. `结果微调`
2. `结果问题`
3. `实施结果`

我目前最倾向：

1. `结果微调`

原因是：

1. 和你这轮需求主题一致。
2. 语义明确。
3. 用户看到就知道这个目录是干什么的。

## 我当前判断

现在这部分需求已经从“是否要复用流程”收敛到了“如何把进入和返回做得最轻”。

我目前认为最稳的一版是：

1. 主目录态显示 `建问题`
2. 子目录态显示 `返`
3. 子目录状态通过路径结构判定
4. 状态标记优先用短文字或简单符号，不上重图标方案
5. 结果微调总目录名固定后，就可以进入完整界面方案讨论
