# v6

这轮之后，大方向基本已经都定下来了。

现在剩下的最后一个小点，就是：

`子目录状态标记，到底用哪个短符号最合适。`

你已经明确了三件事：

1. 主按钮名就用 `建问题`
2. 结果微调总目录名就用 `结果微调`
3. 状态标记可以用“小符号 / Unicode 图形字符”这一路

所以这版我就不再继续讨论流程模型，只把这个状态标记问题收掉。

## 我对状态标记的要求判断

这个标记要成立，我认为至少要满足 4 个条件：

1. `足够短`
   当前 UI 空间很紧，不能占宽。

2. `足够醒目`
   要让用户一眼知道“现在不是主目录，是问题子目录”。

3. `直觉上不容易误解`
   最好一看就像“标记 / 提醒 / 分支 / 子项”之类的意思。

4. `在当前技术栈里稳定`
   不能选那种很可能字体不支持、显示成方块、或者在不同机器上表现不稳的特殊字符。

## 我筛过之后的几类候选

### 1. 纯文字类

例如：

1. `子`
2. `微`
3. `题`

优点：

1. 最稳。
2. 基本不会乱码。
3. 语义明确。

缺点：

1. 视觉提醒感不够强。
2. 更像标签，不像状态标记。

如果要极稳，`子` 是能用的，但我觉得还不够“醒目”。

### 2. 普通几何符号类

例如：

1. `•`
2. `●`
3. `◆`
4. `◇`
5. `■`
6. `□`

优点：

1. 很短。
2. 很容易做出“这里有状态”的视觉提醒。
3. 字体兼容性通常不错。

缺点：

1. 单独看语义偏弱。
2. 用户第一次看到时，需要习惯一下它代表“子目录态”。

这类里我觉得 `◆` 和 `●` 比 `•` 更醒目。

### 3. 箭头/分支感符号类

例如：

1. `↳`
2. `→`
3. `↪`

优点：

1. 有“进入子层 / 分支出去”的感觉。
2. 和“当前处于子问题目录”这个语义比较贴近。

缺点：

1. 某些符号字体表现不一定完全一致。
2. 有些箭头更像动作，不像状态。

这里面 `↳` 语义其实挺贴切，因为它天然有“下挂子项”的感觉。

### 4. 特殊图形字符类

例如：

1. `※`
2. `✦`
3. `✳`
4. `⚑`

优点：

1. 很醒目。

缺点：

1. 容易偏装饰化。
2. 与当前朴素工具风格不一定协调。
3. 部分字符在不同字体下稳定性不如普通几何符号。

我不太建议走这类，容易显得用力过猛。

## 我当前最推荐的候选

如果现在就要我定优先级，我会给下面这个排序：

### 第一推荐：`↳`

我最推荐这个。

原因：

1. 它很短。
2. 它有很强的“子层 / 下挂分支”直觉。
3. 放在当前目录名旁边时，很容易表达“当前是从主题目录派生出来的问题子目录”。
4. 视觉上比纯文字更醒目，但又不浮夸。

它比较适合当前这个需求的语义。

### 第二推荐：`◆`

如果你更想要一个更稳定、更像“状态点”的符号，那我第二推荐 `◆`。

原因：

1. 足够醒目。
2. 几何形状清楚。
3. 通常兼容性不错。

缺点是它更像“有状态”，但不如 `↳` 那么贴近“子目录”语义。

### 第三推荐：`子`

如果你想最大限度追求稳，不追求图形感，那就用 `子`。

它最稳，但视觉提醒感比前两个弱一些。

## 我当前最倾向的具体做法

我现在最倾向：

1. 默认方案用 `↳`
2. 如果实际显示出来在当前字体下不够好看或对齐不稳，再退到 `◆`
3. 如果你极度保守，最后再退到 `子`

也就是说，我建议把它当成一个三档方案，而不是一开始就选最保守的。

## 状态标记的放置判断

我顺手把这个点也一起收一下。

我认为这个标记最适合：

1. 放在当前目录显示区域附近
2. 只在问题子目录状态下显示
3. 主目录状态下隐藏

这样它不会额外占一整块按钮位，而只是一个轻量提示。

这和你前面要的“精简”是完全一致的。

## 我现在确认下来的关键口径

到这一轮为止，我认为已经可以把这部分需求的关键口径收成下面这样：

1. 结果微调总目录固定名为 `结果微调`
2. 主目录状态下显示 `建问题`
3. 问题子目录状态下显示 `返`
4. `建问题` 只在主题目录可用，并负责创建/进入下一个两位编号子目录
5. `返` 只在问题子目录可用，并直接回到主题目录
6. 子目录状态通过路径结构判定：
   - 当前目录名为两位数字
   - 父目录名为 `结果微调`
7. 子目录状态标记优先使用短符号
8. 当前最推荐的状态标记是 `↳`
