# V1 想法与问题整理

## 我对当前需求的理解

本次需求是在现有 AIProcess 工具基础上进行两项调整：

1. **去掉**现有的【先别改】按钮。
2. **新增**一个【复执行】按钮，并附带一个复选框【步】。

这个新按钮服务于"正式实施阶段"——即当需求沟通已收敛到 `实施文档.md` 后，用户需要让 AI 基于该文档进入实际执行阶段。因此它的定位不是"沟通流程"内的动作，而是沟通完成后"启动执行"的动作。

---

## 对现有代码的理解

### 核心架构

- **技术栈**：AutoHotkey v2.0，单文件脚本 `app/AIProcess.ahk`。
- **配置持久化**：`app/config/settings.ini`，使用 `IniRead`/`IniWrite` 读写。
- **模板系统**：`app/templates/` 目录下存放固定提示词模板，支持运行时读取和变量替换（如 `{{filePath}}`）。
- **界面风格**：轻量置顶悬浮窗，宽度约 210px，Microsoft YaHei UI 字体，按钮成对横向排列（宽 60px，间距 6px）。
- **全局状态**：`CurrentDir` 决定所有按钮的可用性；`AppConfig` 映射存储运行期配置。

### 当前按钮布局（从上到下）

| 行 | 控件 | 说明 |
|---|---|---|
| 1 | 当前目录显示 + 设目录 | 状态与目录设置 |
| 2 | 用IDEA开 + 禁改正式 | 复选框开关 |
| 3 | 建需求 + 复需求 | 需求阶段动作 |
| 4 | 建回复 + 复回复 + 实 | 回复阶段动作（"实"是复选框） |
| 5 | 复关系 + 先别改 | 辅助动作 |

### 关键函数与模式

- `LoadTemplate(fileName)`：从 `templates/` 读取 UTF-8 文本。
- `AppendNoModifyPromptIfNeeded(content)`：根据"禁改正式"复选框状态，在提示词末尾追加约束文案。
- `AppendReplyImplementationTailIfNeeded(content)`：根据"实"复选框状态，在回复提示词末尾追加"如果没有新的问题，请创建 实施文档.md"。
- `SetControlsEnabled(enabled)`：统一控制所有按钮的可用状态。
- `EnsureCurrentDirectory()`：校验 `CurrentDir` 是否已设置。

## 新的想法

### 1. 【复执行】的提示词建议拆成两个模板

参考现有模板系统（`requirement_prompt.txt`、`reply_prompt.txt` 等），建议为【复执行】增加独立的模板文件，而非硬编码在脚本中。

理由：

- 保持与现有架构一致，降低维护成本。
- 用户后续可能会调整"执行阶段"的提示词语气或约束条件。

建议拆成两个模板：

- **`execute_prompt.txt`**：不勾选【步】时使用。核心语义是"请查看实施文档.md，并立刻执行"。
- **`execute_step_prompt.txt`**：勾选【步】时使用。核心语义是"请查看实施文档.md，判断是否需要先拆解成步骤目录，还是先直接执行"。

### 2. 界面布局需要微调

当前窗口宽度为 210px，按钮宽 60px × 2 + 间距 6px = 126px，刚好容纳一对按钮。

如果【复执行】与【步】复选框放在同一行：

- 【复执行】宽 60px + 间距 6px + 【步】复选框宽约 30px = 96px，**可以放下**。
- 去掉【先别改】后，第 5 行直接替换为【复执行 + 步】，无需额外增加行高。

但有一个细节：现有代码中 `actionButtonWidth := 60`，复选框使用 `x+gap yp+4` 进行微调。如果【步】复选框文字只有一个字，实际渲染宽度较小，需要测试是否会导致视觉上不整齐。

### 3. 【步】复选框的状态建议持久化

参考"用IDEA开"和"禁改正式"两个复选框，它们的状态都通过 `IniWrite` 持久化到 `settings.ini` 中。建议【步】也采用同样的模式：

- 增加 `AppConfig["ExecuteStepMode"]` 配置项。
- 增加 `settings.ini` 中 `[Behavior]` 或 `[Prompt]` 段下的 `ExecuteStepMode` 键值。
- 在 `LoadConfig()` 和对应 Toggle 函数中处理读写。

理由：用户的执行习惯（偏好"直接执行"还是"先拆步骤"）通常是稳定的，不需要每次手动重新勾选。

### 4. 【复执行】的可用性规则建议增加前置校验

现有按钮的可用性分两层：

1. 未设置目录时，全部禁用。
2. 已设置目录后，全部启用（部分按钮在点击后内部再做文件存在性校验，如"建回复"会检查是否有 vX.md）。

建议【复执行】在已设置目录的基础上，**额外检查 `实施文档.md` 是否存在**：

- 如果不存在，点击时提示"当前目录下未找到 实施文档.md"，不复制到剪贴板。
- 这样避免用户在没有实施文档的情况下误操作。

另一种方案是：按钮始终可用，但复制的内容中文件路径指向一个不存在的位置，依赖 AI 反馈报错。但前者体验更好，且与"建回复"的逻辑一致（没有 vX.md 时报错）。

### 5. 去掉【先别改】后，其原有功能是否完全废弃

现有【先别改】按钮的功能非常简单：

```autohotkey
CopyDiscussPrompt(*) {
    A_Clipboard := "先别改，谈谈"
    ShowFeedback("已复制：先别改，谈谈")
}
```

它只是一个纯文本复制按钮，没有任何模板文件依赖。去掉后：

- UI 上减少一个按钮，空间更干净。
- 但用户如果偶尔还需要"先别改，谈谈"这句话，需要手动输入。

建议：如果完全去掉，需要确认用户是否接受这一行为。或者可以考虑将"先别改，谈谈"作为一个更通用的"暂停执行"提示词，合并到【复执行】的某种状态里？不过需求原文明确说"去掉"，这里不做过度设计，仅做确认。

## 我认为当前需求里还缺的关键问题

### 1. 【复执行】不勾选【步】时，提示词的精确语义是什么

需求原文说："表示立刻根据实施文档.md执行"。

但"执行"在 AI 协作语境中有歧义：

- 是指"直接开始修改代码、创建文件、执行实际操作"？
- 还是指"阅读实施文档.md 后，给出执行计划，但等用户确认后再动手"？
- 是否需要保留"禁改正式"的约束？（即在执行阶段，是否还需要"不要修改正式代码"的提示？）

**建议明确**：不勾选【步】时的提示词，除了告知 AI 查看 `实施文档.md`，是否还需要附加约束（如"按文档执行，直接修改代码"或"先输出执行计划"）。

### 2. 【复执行】勾选【步】时，询问的精确话术是什么

需求原文说："表示询问，即询问模型是否需要先根据实施文档.md创建一个【实施步骤】的目录，按照合理的步骤顺序，为每个步骤创建一个md文件放在实施步骤目录下。还是可以直接执行。"

这里有几种可能的提示词写法：

- **方案 A**："请查看实施文档.md，判断是否需要先将其拆解为多个步骤。如果需要，请创建'实施步骤'目录，并在其中按顺序创建每个步骤的 md 文件；如果不需要，请直接执行。"
- **方案 B**："请查看实施文档.md，先创建'实施步骤'目录并生成步骤文件，生成后询问我是否满意，满意后再执行。"
- **方案 C**："请查看实施文档.md，如果内容较复杂，建议先拆分为实施步骤；如果内容简单，可直接执行。请告诉我你的判断和下一步计划。"

**建议明确**：用户希望 AI 是"自主判断并执行"，还是"先给出建议，等用户确认后再执行"。这会影响提示词的语气和结构。

### 3. "实施步骤"目录的文件命名规则

参考基础需求的已有实践，基础需求的 `实施步骤` 目录下文件名为：

```
步骤1-项目骨架与配置.md
步骤2-窗口托盘与快捷键.md
步骤3-目录与文件功能.md
...
```

本次需求是否沿用相同的命名规则？还是只需要 AI "自行判断合理顺序"，不强制命名格式？

**建议明确**：工具层面是否需要对"实施步骤"目录做扫描、识别、校验？还是完全交由 AI 自主决定目录名和文件名？

### 4. 【复执行】是否受"禁改正式"复选框影响

现有"禁改正式"复选框会在所有提示词末尾追加约束文案："当前阶段不要修改正式代码，不要修改已有正式文件"。

但【复执行】按钮的场景是"正式进入实施阶段"，此时用户的目的恰恰是**希望** AI 修改代码。如果"禁改正式"仍然勾选，追加的约束会与【复执行】的目的冲突。

**建议明确**：

- 【复执行】复制的提示词是否**忽略**"禁改正式"的追加？
- 还是保持现有逻辑（仍然追加），由用户手动关闭"禁改正式"后再使用【复执行】？
- 或者增加一个联动逻辑：点击【复执行】时自动关闭"禁改正式"？

### 5. 窗口高度是否需要调整

现有窗口高度为 182px（由 `settings.ini` 中 `Window/Height` 控制）。去掉【先别改】并增加【复执行 + 步】后：

- 按钮行数不变（第 5 行替换而非新增）。
- 但复选框【步】的高度可能与按钮略有差异，需要确认最终渲染后的总高度是否仍然合适。

**建议**：在实现后根据实际界面微调窗口默认高度，或在代码中适当放宽自适应空间。

## 我目前最想确认的问题

1. 【复执行】不勾选【步】时，提示词除了"请查看实施文档.md 并立刻执行"，还需要附加哪些约束或说明？
2. 【复执行】勾选【步】时，你希望 AI 是"自主判断后直接执行或拆步骤"，还是"先给出建议，等用户确认后再执行"？
3. "实施步骤"目录及内部文件的命名，是否需要工具层强制规范（如沿用"步骤N-标题.md"），还是完全交给 AI 自行决定？
4. 使用【复执行】时，现有的"禁改正式"约束是否应当自动忽略或自动关闭？
5. 如果当前目录下不存在 `实施文档.md`，【复执行】按钮点击后应该提示错误（与"建回复"逻辑一致），还是仍然复制提示词（让 AI 自行处理）？

## 结论

本次需求的核心是**在"沟通流程"结束后，增加一个"启动执行"的快捷入口**。这是一个合理的功能延伸，因为现有工具已经完整覆盖了从 `需求.txt` → `vX.md` → `实施文档.md` 的沟通链路，唯独缺少从 `实施文档.md` 进入实际执行的触发动作。

实现上需要注意的关键点：

1. **保持架构一致性**：复用现有的模板系统、配置持久化机制和 UI 布局模式。
2. **处理好边界冲突**：特别是"禁改正式"约束与"复执行"目的之间的潜在矛盾。
3. **明确提示词语义**：这直接决定按钮的实用价值，需要用户确认后再定稿模板内容。
4. **界面空间评估**：当前窗口较窄，新增复选框后需测试实际渲染效果，必要时微调默认窗口尺寸。

如果以上 5 个确认问题能明确，后续可以直接进入 `实施文档.md` 的编写。
