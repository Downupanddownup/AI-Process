# V2 想法与问题整理

## 对 V1 回复的确认与消化

### 1. 布局方案——已明确

参考截图和回复，【复执行】与【步】复选框将完全复用现有【复回复】+【实】的横向布局模式：

- 按钮宽 60px + 间距 6px + 复选框
- 行数不变（第 5 行替换【先别改】）
- 窗口高度无需调整

### 2. 前置校验——已确认

【复执行】将在已设置目录的基础上，额外检查 `实施文档.md` 是否存在。不存在时提示"当前目录下未找到 实施文档.md"，不执行复制。

### 3. 【先别改】——已确认废弃

该按钮及其背后的 `CopyDiscussPrompt` 函数将完全移除，不再保留任何备用入口。

### 4. 不勾选【步】时的提示词语义——已明确

> 请你根据当前的实施文档，修改正式代码和文件，完成整个方案的落地。

关键点：**不需要追加"禁改正式"约束**。这意味着【复执行】的提示词生成逻辑需要独立于 `AppendNoModifyPromptIfNeeded`，即使"禁改正式"复选框处于勾选状态，也不影响【复执行】的复制内容。

### 5. 勾选【步】时的行为——已明确

执行逻辑分两支：

1. **不需要拆解**：直接修改正式代码和文件，完成方案落地。
2. **需要拆解**：
   - 在当前目录下创建【实施步骤】目录
   - 模型自主判断步骤数量，按 `01-xxx.md`、`02-xxx.md` 格式命名
   - 每个步骤文档包含标题和顺序
   - 步骤文档写完后**暂停**，等待用户验收，不直接修改代码

### 6. 实施步骤目录的命名规则——已明确

工具层不干预命名格式，完全交由 AI 自主决定。唯一约束：

1. 文件有先后顺序
2. 格式为 md 文档
3. 整体包含实施文档的全部内容

### 7. 与"禁改正式"的关系——已明确

【复执行】**不受**"禁改正式"复选框影响。这是产品层面的正确设计——进入执行阶段的目的就是修改代码，不应再受沟通阶段的约束干扰。

## 新的想法

### 1. 建议为【复执行】新增两个模板文件

基于已明确的提示词语义，建议按现有模板系统惯例，在 `app/templates/` 下新增：

- **`execute_prompt.txt`**（不勾选【步】时使用）

  内容草案：
  ```
  请你根据当前的实施文档：{{filePath}}，修改正式代码和文件，完成整个方案的落地。
  ```

- **`execute_step_prompt.txt`**（勾选【步】时使用）

  内容草案：
  ```
  请你查看当前的实施文档：{{filePath}}。

  请先判断该方案的内容复杂度，是否需要拆解为多个实施步骤：
  - 如果不需要拆解，请直接修改正式代码和文件，完成整个方案的落地。
  - 如果需要拆解，请在当前目录下创建【实施步骤】目录，按照合理的执行顺序为每个步骤创建 md 文件（格式如 01-xxx.md、02-xxx.md 等），确保所有步骤合起来覆盖实施文档的全部内容。步骤文档写完后请先停下来，告诉我已经写好，等待我验收。不要在未获得我确认前直接修改正式代码。
  ```

理由：保持与现有 `requirement_prompt.txt`、`reply_prompt.txt` 的架构一致性，后续如需调整话术无需改代码。

### 2. 【步】复选框的状态建议持久化

参考"用IDEA开"和"禁改正式"的实现，建议将【步】的勾选状态持久化到 `settings.ini`：

- 增加配置项 `[Behavior] ExecuteStepMode`（默认值 `0`，即不勾选）
- 在 `LoadConfig()` 中读取
- 在切换时 `IniWrite` 写入

理由：用户对于"偏好直接执行还是偏好先拆步骤"的习惯通常是稳定的，每次打开工具后保持上次的勾选状态更符合使用直觉。

### 3. 【复执行】的反馈文案建议区分模式

现有按钮点击后都有明确的反馈（如"需求提示词已复制"）。建议【复执行】也做区分：

- 不勾选【步】时："执行提示词已复制"
- 勾选【步】时："步骤模式提示词已复制"

这样用户可以通过反馈文案二次确认当前模式，避免发错提示词。

### 4. 默认配置值的调整

由于去掉了【先别改】按钮并新增了【复执行】，`EnsureDefaultFiles()` 中需要确保新的模板文件在首次运行或缺失时自动创建。建议在初始化逻辑中补充 `execute_prompt.txt` 和 `execute_step_prompt.txt` 的默认生成。

## 结论

本次追加功能的边界已经比较清晰：

1. **去掉**：【先别改】按钮及对应函数。
2. **新增**：【复执行】按钮 + 【步】复选框，共用现有横向布局模式。
3. **核心逻辑**：根据【步】状态复制不同提示词，均指向 `实施文档.md`；不受"禁改正式"影响；缺失实施文档时给出错误提示。
4. **建议补充**：两个模板文件、配置持久化、差异化反馈文案。

如果以上 3 个剩余问题能确认（尤其是模板措辞定稿），后续可以直接进入 `实施文档.md` 的编写。
