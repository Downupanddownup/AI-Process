# V1 想法与问题整理

## 我对当前需求的理解

这个项目的本质，不只是一个“快捷复制话术”的工具，而是一个围绕 AI 协作的轻量级流程启动器。

它主要解决的是以下重复工作：

1. 为一个新主题创建固定结构的目录和文件。
2. 快速生成发给 AI 的标准化提示词。
3. 让 AI 根据当前阶段自动创建 `v1.md`、`v2.md`、`实施文档.md` 等文件。
4. 在上下文切换或新开会话时，快速向新模型说明这些文件之间的关系。

所以从产品定义上看，这个工具更像：

`AI 需求流程快捷面板`

而不是一个普通的文本片段管理器。


## 新的想法

### 1. 不要只做“话术库”，要做“流程动作库”

如果只是管理一堆固定文本，价值会比较有限。更高价值的方式是把每个动作定义成一个“流程按钮”。

例如：

1. `创建需求初始化文件`
2. `生成 V1 提示词`
3. `生成 对V1的回复.txt`
4. `生成 V2 提示词`
5. `生成实施文档提示词`
6. `新会话上下文重建提示词`

每个按钮不只是复制一段文字，而是可以组合执行：

1. 自动获取当前目录路径
2. 自动创建目标文件
3. 自动生成提示词
4. 自动复制到剪贴板

这样你的操作就从“手动拼装流程”变成“点击动作直接执行”。


### 2. 可以把项目抽象成“模板驱动”

你的流程其实很固定，后续很可能不止一种。

例如未来可能会有：

1. `需求澄清流程`
2. `Bug 排查流程`
3. `新会话接管流程`
4. `代码审查流程`
5. `文章整理流程`

所以建议一开始就把“按钮”背后的定义做成模板配置，而不是全写死在代码里。

例如每个模板可以包含：

1. 模板名称
2. 适用场景
3. 需要创建的文件列表
4. 输出提示词模板
5. 是否自动复制
6. 是否需要检查当前目录中已有文件

这样后续扩展成本会低很多。


### 3. 文件命名规则最好交给系统自动判断

你现在已经有明显的版本链条：

1. `需求.txt`
2. `v1.md`
3. `对V1的回复.txt`
4. `v2.md`
5. `对V2的回复.txt`
6. `实施文档.md`

这个版本规则完全适合自动化。

理想行为应该是：

1. 系统扫描当前目录
2. 判断当前已经进行到哪一轮
3. 给出下一步建议动作
4. 一键生成下一阶段文件或提示词

例如当目录里已经有 `v1.md` 但没有 `对V1的回复.txt` 时，面板应该直接高亮：

`下一步：创建 对V1的回复.txt`

这样工具就不只是“快捷入口”，而是带一点“流程感知”。


### 4. 顶部小窗建议做成“命令面板”而不是传统窗口

你强调了：

1. 顶层显示
2. 占比很小
3. 信息密度尽可能高
4. 通过快捷键或搜索唤起

这说明它更像：

`Raycast / PowerToys Run / Alfred` 风格的命令面板

而不是一个常驻大面板。

建议优先考虑两种模式：

1. `搜索执行模式`
   直接输入关键词，例如“v1”“回复”“实施文档”“新会话”，回车执行。
2. `固定动作模式`
   展示少量高频按钮，例如“初始化需求”“生成V1提示词”“新会话接管”。

这两种模式可以共存。


### 5. 增加“安全模式”很有必要

你已经提到了一个关键诉求：

`在需求没有沟通完成之前，禁止修改实际代码和正式文件`

这个点非常重要，建议正式产品化成一个显式选项。

比如：

1. `需求阶段`
   只允许创建讨论文件、复制提示词、整理上下文。
2. `实施阶段`
   才允许生成实施文档、执行代码相关动作。

这能避免 AI 因为提示词不稳定而越权行动。


### 6. 可以增加“上下文打包提示词”

你有一个很实用但容易被忽略的场景：

`旧会话结束后，让新会话快速接管`

这个动作非常值得做成单独按钮，因为它出现频率高，而且模板化价值很强。

这个按钮可以自动做几件事：

1. 扫描当前目录中的关键文件
2. 自动列出文件顺序
3. 自动生成一段“请先重建上下文，再继续后续任务”的提示词

甚至后续可以扩展成：

1. 自动拼接文件路径列表
2. 自动按顺序说明这些文件的用途
3. 自动提醒新模型当前禁止事项


## 我认为当前需求里还缺的关键问题

### 1. 当前目录如何确定

这是最关键的问题之一。

你的所有流程都依赖“当前主题目录”，但目前还没有明确这个目录是怎么被系统识别的。

可能方案有：

1. 手动选择目录
2. 读取资源管理器当前选中文件夹
3. 从剪贴板路径推断
4. 固定监听某个工作区根目录
5. 记录最近使用目录

如果这个问题不明确，后面很多自动化会卡住。


### 2. 这个工具是纯本地助手，还是要和某个 AI 客户端深度配合

需要尽快明确边界：

1. 只是帮你创建文件和复制提示词
2. 还是要进一步帮你把提示词直接发送给某个 AI 客户端
3. 还是只负责“组织内容”，不负责“发消息”

我建议第一版先只做：

`本地文件 + 提示词 + 剪贴板`

不要一开始就做 AI 客户端自动发送，否则复杂度会上升很多。


### 3. 是否需要多套提示词风格

不同 AI 模型听话程度不同。你自己也提到了这一点。

那就会带来一个实际问题：

是否要支持同一个动作下的多种提示词版本？

例如：

1. `简洁版`
2. `强约束版`
3. `高执行力版`

如果这个能力有必要，模板系统就应该预留。


### 4. 文件创建规则要不要允许自定义

你当前举的例子已经很清楚，但未来可能变化。

例如：

1. `需求.txt` 是否固定叫这个名字
2. `对V1的回复.txt` 是否允许改成更短名称
3. `实施文档.md` 是否可能拆成 `实施计划.md` 和 `执行记录.md`

如果第一版全写死，后面重构概率会比较高。


### 5. 你希望“自动化程度”到哪一步

这个问题决定产品设计。

自动化可以分为三层：

1. `只复制提示词`
2. `复制提示词 + 自动创建文件`
3. `自动识别当前阶段 + 自动推荐下一步 + 自动创建文件`

我判断你真正要的是第 3 层，但建议第一版先落第 2 层，保留向第 3 层扩展的结构。


### 6. 是否需要操作历史或撤销能力

如果工具支持一键创建多个文件、覆盖模板内容，那就需要考虑：

1. 是否显示本次创建了哪些文件
2. 是否允许撤销最近一次创建
3. 如果文件已存在，是覆盖、跳过，还是另存为

这些属于很现实的边界问题，最好提前定义。


## 我建议的第一版范围

为了避免第一版做得太散，我建议先只做下面这些：

### 必须有

1. 全局快捷键唤起
2. 小型置顶命令面板
3. 主题目录选择或识别
4. 固定模板的文件创建
5. 固定提示词一键复制
6. 剪贴板写入
7. 基于当前目录检测下一步动作


### 暂时不要做

1. 自动向 AI 客户端发消息
2. 太复杂的 UI
3. 云同步
4. 多人协作
5. 深度编辑器插件集成


## 一个更适合落地的产品结构

我建议界面上只有三块信息：

### 1. 当前目录

显示当前正在操作的主题目录。


### 2. 下一步建议

例如：

`检测到当前已有 需求.txt 和 v1.md，建议下一步：创建 对V1的回复.txt`


### 3. 动作列表

例如：

1. `初始化需求目录`
2. `生成V1提示词`
3. `创建对V1回复文件`
4. `生成V2提示词`
5. `生成实施文档提示词`
6. `生成新会话接管提示词`

这样信息结构会非常干净，也符合你说的“小而高密度”。


## 我目前最想确认的问题

1. 你希望这个工具运行在 Windows 桌面端，还是网页端？
2. “当前主题目录”你希望怎么选中或识别？
3. 第一版是否只做“文件创建 + 提示词复制 + 下一步推荐”，暂时不接 AI 客户端自动发送？
4. 你的固定文件命名规则，是否已经基本稳定？
5. 当目标文件已存在时，你希望默认行为是跳过、覆盖，还是弹出选择？
6. 你希望这个工具主要通过“搜索命令”操作，还是“固定按钮”操作？


## 结论

这个需求是成立的，而且不只是一个“提示词收藏器”，它更接近一个面向 AI 协作流程的微型操作系统。

如果方向按这个思路收敛，第一版最重要的不是 UI，而是三件事：

1. 目录识别
2. 模板系统
3. 流程阶段判断

这三件事一旦定好，后面无论是扩展动作、提示词、文件规则，还是接入更多 AI 使用场景，都会顺很多。
