## 二、对 v4 回复的关键结论确认

### 1. 仓库持久化
- 持久化文件至少保存两个字段：
  - **仓库名称**
  - **仓库加入时间**
- 另外还需要保存仓库路径，否则无法定位仓库。

关于文件名，你提到"用'保存'吧"，我没有完全确定你的具体意思。我理解为：**持久化文件采用 JSON 格式，文件名用 `repositories.json`**。

建议最终结构：

```json
[
  {
    "name": "项目A",
    "path": "D:\\项目A\\需求",
    "addedTime": "2025-07-02 14:30:00"
  },
  {
    "name": "项目B",
    "path": "E:\\项目B\\need",
    "addedTime": "2025-07-02 15:00:00"
  }
]
```

文件位置建议：`history/repositories.json`

如果"保存"是指其他具体文件名（例如 `save.json`、`保存.json`），请告诉我，我可以按你的要求命名。

### 2. 添加仓库的校验规则（最终版）
- **无论是「添加仓库」还是「设目录」，非标准仓库都不加入仓库列表。**
- 标准仓库定义：路径必须以 `\need` 或 `\需求` 结尾。
- 在「添加仓库」弹窗中，非标准路径直接拒绝。
- 在「设目录」功能中，如果用户设置的目录不是标准仓库结构，只切换当前主题，**不创建仓库**。

### 3. 三级弹窗仓库列表
- 显示**完整仓库路径**列表，不是只显示仓库名。
- 因为弹窗空间足够，可以显示完整路径。
- 选中仓库后**必须点击「确定」按钮**才关闭三级弹窗，防止误触。

### 4. 经验总结窗口中已归档主题
- 可以在目录状态列添加「已归档」标签/标识。
- 归属项目列仍显示原仓库名，不显示「归档」。

## 三、最终实施方案草案

### 阶段 1：仓库模型与持久化
1. 创建 `history/repositories.json`（或你确认的其他名称）。
2. 仓库数据结构：`name`、`path`、`addedTime`。
3. 实现仓库识别：`path` 必须以 `\need` 或 `\需求` 结尾。
4. 实现添加仓库：输入路径 → 校验 → 加入 JSON。
5. 启动时加载仓库列表到内存。

### 阶段 2：历史索引新结构
1. 新路径：`history/themes/{仓库名}/{年}/{月}/YYYY-MM-DD.jsonl`
2. 进入主题时，根据主题路径找到所属仓库，写入对应仓库的日期文件。
3. 保持每行 JSONL 格式不变：`time`、`window`、`theme`、`source`、`path`。

### 阶段 3：主面板「设置主题」弹窗
1. 点击主面板当前主题文字 → 打开二级弹窗。
2. 二级弹窗展示：
   - 当前主题完整路径
   - 当前仓库名称（可点击）
   - 当前仓库下主题目录列表（不含归档）
   - 新主题名输入框 + 新建按钮
   - 确定/取消按钮
3. 点击当前仓库名称 → 打开三级弹窗。
4. 三级弹窗展示：
   - 所有仓库完整路径列表
   - 新增仓库按钮
   - 确定/取消按钮
5. 点击新增仓库 → 打开四级弹窗，输入路径并确认。

### 阶段 4：仓库管理窗口
1. 托盘菜单增加「仓库管理」。
2. 窗口布局：
   - 左侧：仓库完整路径列表
   - 右侧：选中仓库下主题目录列表（含复选框）
   - 底部：「归档选中主题」按钮
3. 归档流程：
   - 勾选主题 → 点击归档 → 二次确认弹窗 → 移动到 `仓库\需求\归档\`。
   - 重名时加 `_2`、`_3` …
   - 检查主题是否被任一窗口使用，若使用中则禁止归档。
   - 归档后同步更新该仓库历史索引中的路径。

### 阶段 5：经验总结窗口兼容
1. 读取 `history/themes/` 下的所有仓库数据。
2. 对于已归档主题，在「目录状态」列显示「已归档」。
3. 归属项目列仍显示原仓库名。

### 阶段 6：迁移脚本
1. 单独提供 `.ps1` 或 `.ahk` 脚本。
2. 将旧 `history/index/*.jsonl` 按仓库拆分迁移到 `history/themes/{仓库名}/{年}/{月}/`。
3. 运行一次后作废，新代码不再读取 `history/index/`。
