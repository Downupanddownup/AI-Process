## 二、对 v2 回复的关键结论确认

以下是我根据回复整理出的最终方案，请确认是否有遗漏或误解。

### 1. 历史索引迁移
- 单独做一个**临时迁移脚本**，将旧 `history/index/` 数据迁移到新结构。
- 迁移完成后，新代码不再读取 `history/index/`，只使用新结构。
- 新目录名称候选：`history/data/` 或 `history/theme/`（待最终确定）。

### 2. 仓库名称冲突
- 仓库名冲突时加序号，例如：`仓库名_02`、`仓库名_03`。
- **不使用仓库 ID**，直接以仓库名称（必要时加序号）作为标识。

### 3. 新增仓库输入路径
- 用户输入的路径**必须以 `\need` 或 `\需求` 结尾**。
- 例如：`D:\项目A\need` 或 `D:\项目A\需求`。

### 4. 仓库名称定义
- 仓库名 = `need` 或 `需求` 前面的目录名。
- 例如 `D:\项目A\需求` → 仓库名为 `项目A`。
- 保留现有「设目录」功能，并扩展仓库检查：
  - 若该主题目录属于已存在仓库，则直接切换主题。
  - 若该主题目录属于新的 `need/需求` 目录结构，则自动创建仓库。
  - 若找不到标准结构，则归入一个**兜底仓库**（名称待确定，例如 `other`）。

### 5. 弹窗层级
- **主面板** → 点击当前主题文字 → 打开**二级弹窗**（展示当前主题路径、当前仓库、仓库下主题列表）。
- **二级弹窗** → 点击当前仓库 → 打开**三级弹窗**（仓库选择、新增仓库入口）。
- **三级弹窗** → 点击新增仓库 → 打开**四级弹窗**（输入路径、确认）。
- 「设目录」按钮保持原有功能，与主题切换弹窗无关。

### 6. 归档规则
- 归档目录：`{仓库路径}\归档\`（例如 `D:\项目A\需求\归档\`）。
- 重名时在主题名后加编号，**从 `_2` 开始**：`主题A_2`、`主题A_3` …
- **不支持取消归档**。

### 7. 归档主题展示
- **经验总结窗口**：展示归档主题，但归属项目列不能显示为「归档」，仍需显示原仓库名。
- **日报/周报**：必须包含归档主题，不能排除。
- **切换主题的快捷列表**（主面板二级弹窗中的主题列表）：不显示已归档主题。

### 8. 多窗口与仓库
- **仓库列表全局共享**。
- 每个窗口的**当前主题独立**。
- 归档时检测：若某主题正被任一窗口使用，则**禁止归档**。

### 9. 仓库持久化
- 仓库数据**不放 `settings.ini`**。
- 字段极简，核心是仓库路径（仓库名可实时从路径提取）。
- 具体持久化文件位置待确定。

## 三、仍需最终确认的细节

### 1. 新历史索引目录名称
- 候选：`history/data/` 或 `history/theme/`。
- 你倾向用哪个？我个人建议用 `history/themes/`（复数），语义更明确，但也可以听你决定。

### 2. 兜底仓库名称
- 对于不符合标准结构的目录，归入哪个仓库？
- 候选：`other`、`未分类`、`default`。
- 是否允许用户手动重命名兜底仓库？

### 3. 仓库持久化文件
- 你同意不放在 `settings.ini`。
- 建议持久化到 `app/config/repositories.txt`（每行一个仓库路径，简单）或 `app/config/repositories.json`（便于后续扩展）。
- 你倾向哪种？
- 如果只用路径一个字段，那 `repositories.txt` 最简单：每行一个以 `\need` 或 `\需求` 结尾的绝对路径。

### 4. 历史索引新结构的具体路径
- 方案 A：`history/themes/{仓库名}/YYYY-MM-DD.jsonl`
- 方案 B：`history/themes/{仓库名}/{年}/{月}/YYYY-MM-DD.jsonl`
- 你倾向哪种？考虑到按仓库归类后数据量不会太大，方案 A 更简单直接。

### 5. 三级弹窗的仓库选择交互
- 三级弹窗里：
  - 是显示仓库**下拉框** + 「新增仓库」按钮？
  - 还是显示仓库**列表** + 底部「新增仓库」按钮？
- 选择仓库后，是立即关闭三级弹窗回到二级弹窗刷新主题列表，还是需要点「确定」？

### 6. 归档操作前的确认
- 批量勾选主题后点击「归档」，是否需要弹出确认框：「确定归档 X 个主题？归档后不可恢复。」？
- 还是直接执行？

### 7. 归档目录的位置
- 你确认归档目录放在 `{仓库路径}\归档\` 下，例如 `D:\项目A\需求\归档\`。
- 是否允许存在多个归档目录？例如 `归档_2`、`归档_3`？
- 还是只有一个固定的 `归档` 目录？

## 四、实施方案预告

确认以上细节后，我将按以下顺序开发：

1. **仓库模型与持久化**：仓库识别、添加、持久化存储。
2. **历史索引新结构**：按仓库分目录写入，同时保留旧结构只读兼容。
3. **主面板主题切换弹窗**：二级弹窗展示当前主题、仓库、主题列表，三级弹窗仓库选择/新增。
4. **仓库管理窗口**：托盘菜单入口，支持归档操作。
5. **归档功能**：移动目录、重名处理、历史索引路径同步、使用中主题保护。
6. **迁移脚本**：一次性将旧 `history/index/` 迁移到 `history/themes/`。
