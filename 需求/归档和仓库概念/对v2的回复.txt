好，接下来我们讨论后面几个问题，关于历史索引的迁移。

1. 历史索引的迁移
   * 迁移策略：这部分很明显到时候会单独作为一个功能，不单独做一个脚本来对当前现有的目录做一次迁移。迁移完这个脚本就可以作废掉了。
   * 目录重命名：在现在的环境下，其实这个东西比较简单。后面索引的话，我换一个新的名称，不叫 index 目录了，换一个比如叫 data 或者叫主题 theme。然后在里面我们再去做归类、做处理。
   * 仓库迁移：或者说你说的仓库 re 之类的玩意儿，这样的话就不影响了。后面我多写个脚本去给它迁过来就行了，理解吧？迁移完了就行了。迁移完成之后这个就没有用了，这玩意儿就只是提供了一个临时脚本就好了。新的代码完全不需要去管它 index 的东西，我们要的就是全新的、完整的。

2. 仓库名称冲突
   * 解决方法：仓库名称冲突的话，你就后面加序号，比如杠 02、03、04，这样肯定就不冲突了。
   * 仓库 ID：所以你说仓库 ID 是哪里来的？没有仓库 ID，什么仓库 ID？

3. 新增仓库时输入路径的精度
   * 精度规则：精度那就是最后必须是以 need 或者需求结尾，就这样，别的就没了。

4. 仓库名称定义
   * 命名逻辑：它的仓库名称就是 need 或需求前面的部分，就叫仓库名称。
   * 关系澄清：需求主题和这有什么关系？需求主题那是主题内，需求文件是主题下的东西，不是仓库下的东西。
   * 功能保留：所以这个功能肯定是要保留的。你为什么要删除它？这个功能本来就肯定要保留的，还是当前的功能，只不过你要给它扩展一下。它不能只是设主题，你还得检查一下它仓库有没有。
     (a) 如果有：按照当前逻辑，如果仓库已经有了，那就不用处理。
     (b) 如果没有：那就给他加一个，先创建一个仓库。
     (c) 判断规则：仓库的判断规则就是当前的规则，即往上的目录层级找。找不到就算了，找不到的话就算没有。
     (d) 抛锚兜底：对于这种没有标准的目录，就单独给他一个兜底选项（比如叫 other 或者叫一个特殊的仓库），把那些没有标准的放进去就行。

5. 二级弹窗新增仓库的交互
   * 流程设计：点击新增仓库后，应该是在当前弹窗内输入。我觉得新建仓库应该放在三级弹窗里面。因为二级弹窗当前其实是当前目录、当前仓库以及那仓库下的列表。当你点击当前仓库时有一个三级弹窗，在三级弹窗里面可以选择点击新增仓库。这时候会有个四级弹窗，四级弹窗就是输入路径然后去确认一下。

6. 归档重命名机制
   * 归档编号：归档一二三四五六七往下走，从二开始吧，二三四五六七。
   * 是否支持取消归档：不支持。这种功能太容易出问题了，不支持。

7. 归档主题在经验总结窗口中的展示
   * 展示要求：归档主题肯定也要展示，都要展示都需要去展示。
   * 注意事项：在显示它归属的一个仓库名称的时候，你记得别把归档当做它的仓库名称。

这个逻辑你要搞清楚的，不要它归档之后，它的仓库名称就变成“归档”，那是不行的。

关于周报、日报生成时是否要排除归档仓库的问题：
1. 肯定不能排除的。如果你排除了，那我周报、日报是不是全没数据了？肯定得包含的。
2. 必须包含多仓库和多窗口的仓库。
3. 仓库列表是贡献所有窗口都贡献。哪个窗口当前主题是个独立的，这种窗口肯定是独立的。从全局窗口可以选择仓库是全局的，但是每个窗口各自的主题是独立的。
4. 在做归档的时候注意一下：如果说某个主题它本身在窗口中是在使用的，那这个就不能归档。
