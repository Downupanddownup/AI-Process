# V4 技术选型与实施计划

## 1. 这一版新增目标

你这次补充的要求，核心不是功能流程，而是工程实现方式要满足下面几点：

1. 提示词内容要容易修改
2. 修改位置要集中，不能分散在代码各处
3. 改完程序内容后，重新启动即可生效
4. 整个程序尽量轻量，资源占用要低

所以这一版的重点不是继续补功能，而是把项目的技术实现原则定清楚。


## 2. 技术目标重新定义

这个项目第一版最适合做成：

`一个轻量级、本地化、配置驱动的 Windows 桌面工具`

这里有三个关键词：

1. `轻量级`
   启动快，常驻资源占用低
2. `本地化`
   所有逻辑和提示词都放在本地文件，不依赖云端
3. `配置驱动`
   提示词、按钮说明、文件关系说明等内容尽量从配置文件读取，而不是硬编码在多个代码文件里


## 3. 我建议的总体实现思路

第一版建议采用：

`界面代码 + 本地配置文件 + 少量文件操作逻辑`

也就是说，把系统拆成三层：

### 3.1 界面层

只负责：

1. 显示当前目录
2. 显示按钮
3. 响应点击
4. 显示提示信息
5. 管理 IDEA 开关状态

界面层尽量不要直接写提示词正文。


### 3.2 配置层

专门负责存放：

1. 需求文件提示词
2. 回复文件提示词
3. 文件关系说明文案
4. 按钮标题
5. 提示信息文案

你的高频修改内容，应该尽量都放在这一层。


### 3.3 逻辑层

只负责：

1. 路径校验
2. 文件创建
3. 扫描 `vX.md`
4. 识别最新版本
5. 拼接最终复制文本
6. 调用 IDEA 打开文件

逻辑层尽量只处理规则，不直接写大段中文提示词。


## 4. 最关键的设计原则：提示词必须集中管理

这是你这次最重要的诉求。

我建议第一版必须采用：

`提示词配置文件集中维护`

不要把这些内容散落在：

1. 按钮点击事件里
2. 多个组件文件里
3. 多个工具函数里
4. 条件分支内部

正确做法应该是：

1. 建一个专门的配置目录
2. 把所有提示词都收进 1 到 2 个配置文件
3. 程序启动时读取这些配置
4. 你修改配置文件后，重启程序即可生效


## 5. 我建议的目录结构

建议项目结构尽量简单：

```text
project/
  src/
    main/
      app
      fileService
      promptService
      ideService
    ui/
      window
      components
  config/
    prompts.json
    ui.json
  assets/
  package.json
```

这里重点是 `config/` 目录。


## 6. 配置文件应该怎么设计

### 6.1 `prompts.json`

这个文件专门存放提示词正文。

建议至少包含：

1. `requirementPrompt`
2. `replyPrompt`
3. `contextRelationPrompt`

示意结构：

```json
{
  "requirementPrompt": "这是一个需求文件，请你查看这个文件：{{filePath}}。如果你有新的想法或问题，请写入 v1.md 或下一版本文档；如果没有新的问题，请创建实施文档.md。",
  "replyPrompt": "这是一个对当前版本文档的回复文件，请你查看这个文件：{{filePath}}。如果你还有新的想法或问题，请继续创建下一版 vX.md；如果没有新的问题，请创建实施文档.md。",
  "contextRelationPrompt": "当前目录中的需求.txt、vX.md、对vX的回复.txt、实施文档.md 共同构成完整需求沟通过程。请按时间顺序阅读这些文件，并优先完成上下文重建。"
}
```

说明：

1. 文本里的变量采用占位符方式，例如 `{{filePath}}`
2. 程序运行时再把真实路径替换进去

这样后续你只需要改这个文件，不需要到代码里翻。


### 6.2 `ui.json`

这个文件专门存放界面文案和提示信息。

建议包含：

1. 按钮名称
2. 错误提示
3. 成功提示
4. 占位文本

示意结构：

```json
{
  "buttons": {
    "setPath": "定位切换",
    "createRequirement": "创建需求.txt",
    "copyRequirementPrompt": "复制需求文件提示词",
    "createReply": "创建对vX的回复.txt",
    "copyReplyPrompt": "复制回复文件提示词",
    "copyRelations": "复制文件关系说明"
  },
  "messages": {
    "pathRequired": "请先设置当前主题目录",
    "versionFileMissing": "当前目录下未找到 vX.md 文件",
    "fileExists": "文件已存在",
    "pathInvalid": "路径不是有效目录"
  }
}
```

这个文件的价值是：

后续如果你想改按钮文案或提示语，也不需要改代码。


## 7. 技术选型建议

你要求程序轻量、桌面可用、重启即可加载本地配置。

基于这个目标，我建议优先考虑：

`Tauri + 前端轻界面`

### 为什么优先考虑 Tauri

优点：

1. 相比 Electron 更轻
2. 内存占用通常更低
3. 启动速度更有优势
4. 很适合这种本地小工具
5. 文件操作、剪贴板、窗口控制都能做
6. 本地读取 JSON 配置很方便

这类项目如果用 Electron 也能做，但对你这个需求来说偏重了。


## 8. 前端界面技术建议

界面层建议保持简单，不要引入太重的 UI 框架。

推荐方向：

1. `Tauri + Vanilla HTML/CSS/TypeScript`
2. 或 `Tauri + Preact`

我的倾向是：

`第一版优先 Tauri + Vanilla TS`

理由：

1. 功能简单，页面不复杂
2. 不需要复杂状态管理
3. 不需要组件体系撑场面
4. 包体和运行开销都更可控
5. 代码文件数量不会太多

如果一开始就上 React，当然也能做，但对第一版来说有点偏重。


## 9. 为什么不建议把提示词直接写进前端组件

如果提示词直接写进按钮点击逻辑里，会很快出现几个问题：

1. 你不知道该去哪个文件改
2. 同一种文案可能会复制多份
3. 后续加一个提示词就要动代码
4. 中文内容和逻辑代码混在一起，会越来越乱

所以第一版一定要坚持一个原则：

`界面只调用 prompt key，不直接存 prompt 正文`

例如：

1. 点击“复制需求提示词”
2. 界面只调用 `promptService.getRequirementPrompt(filePath)`
3. `promptService` 去配置文件里读取 `requirementPrompt`
4. 再替换 `{{filePath}}`

这样结构就干净了。


## 10. 程序启动与生效机制

你要求的是：

`退出程序 -> 改内容 -> 再启动 -> 生效`

这个目标非常适合配置文件方案。

推荐机制：

1. 程序启动时读取 `config/prompts.json`
2. 同时读取 `config/ui.json`
3. 运行过程中配置内容驻留在内存
4. 你修改配置文件后，不需要重新打包
5. 只要退出程序并重新启动，就重新加载新配置

如果后面你想进一步升级，也可以支持“点击刷新配置”。

但第一版没必要做热更新，重启生效已经够用了。


## 11. 文件与服务职责建议

为了避免后续维护混乱，我建议职责划分尽量固定。

### 11.1 `fileService`

负责：

1. 校验目录
2. 创建文件
3. 扫描目录中的 `vX.md`
4. 获取最新版本号
5. 拼接目标文件路径


### 11.2 `promptService`

负责：

1. 读取 `prompts.json`
2. 替换变量
3. 输出最终复制文本


### 11.3 `uiConfigService`

负责：

1. 读取 `ui.json`
2. 提供按钮标题和提示语


### 11.4 `ideService`

负责：

1. 调用 IDEA 打开指定文件


这样以后你想改提示词，只碰配置文件。
你想改文件规则，只看 `fileService`。
你想改 UI 文案，只改 `ui.json`。


## 12. 第一版实施计划

### 阶段 1：搭建空壳

目标：

1. 建立 Tauri 项目
2. 做出顶部小窗
3. 做出当前目录输入框
4. 做出按钮区和 IDEA 开关

验收：

1. 程序能启动
2. 界面结构完整


### 阶段 2：接入配置系统

目标：

1. 建立 `config/prompts.json`
2. 建立 `config/ui.json`
3. 程序启动时自动读取配置

验收：

1. 修改配置文件后，重启程序可看到文案变化


### 阶段 3：完成文件操作逻辑

目标：

1. 目录校验
2. 创建 `需求.txt`
3. 扫描 `vX.md`
4. 自动生成 `对vX的回复.txt`

验收：

1. 文件能正确创建
2. 文件已存在时能给出正确提示


### 阶段 4：完成提示词复制逻辑

目标：

1. 复制需求文件提示词
2. 复制回复文件提示词

验收：

1. 剪贴板内容正确
2. 变量替换正确

目标：

1. 根据开关决定是否打开文件

目标：

1. 长路径省略显示
2. 鼠标悬停显示完整路径

4. 错误提示优化


## 13. 第一版最重要的工程原则

这一版实现时，必须坚持下面几个原则：

1. 提示词集中放在配置文件
2. UI 文案集中放在配置文件
3. 中文正文不要散落在业务代码里
4. 文件规则集中写在 `fileService`
5. 单个按钮逻辑不要写成长篇点击事件
6. 第一版优先稳定和轻量，不追求功能扩张


## 14. 最终建议

如果按你的目标来做，我认为第一版最合适的路线是：

`Tauri + Vanilla TypeScript + JSON 配置文件`

这是目前最符合你诉求的组合：

1. 足够轻
2. 足够简单
3. 易于本地维护
4. 提示词修改集中
5. 改完配置后，重启即可生效

如果后续功能真的变复杂，再考虑把界面层升级为更强的框架。但第一版没必要把工程做重。


## 15. 结论

这类工具真正难的不是功能，而是“后续怎么维护才不乱”。

所以第一版最重要的技术决策，不是 UI 框架本身，而是：

1. 提示词配置集中化
2. 文案配置集中化
3. 文件逻辑与界面逻辑分离

只要这三件事从一开始就做对，后面你改提示词、改提示语、改流程文案，都会非常省事。
