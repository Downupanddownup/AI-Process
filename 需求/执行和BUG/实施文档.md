# 执行和BUG - 实施文档

## 1. 背景与目标

当前 `AI Process` 的 `复执行` 功能，使用的是：

1. 一个 `复执行` 按钮。
2. 一个 `步` 复选框。
3. 两份执行类模板：
   - `execute_prompt.txt`
   - `execute_step_prompt.txt`

这套结构能表达的只有二元分支：

1. 直接执行。
2. 先判断是否拆步骤，并在需要时生成步骤文档后停下。

但实际使用中已经存在第三种稳定工作流：

1. 不是只生成步骤文档。
2. 而是要生成“步骤目录 + 目录内同名步骤 md 文档”的结构。
3. 该结构生成后停下，后续再围绕各目录继续细化讨论。

当前工具没有直接支持这条工作流，用户只能：

1. 先走“步骤文档”模式。
2. 再手动或再次要求 AI 把步骤文档改造成目录结构。

本次实施的目标，是把这条真实存在的工作流纳入一等能力，并让 `复执行` 的策略表达清晰、可维护、可扩展，同时保持现有小面板工具的风格和代码组织方式。

## 2. 需求边界

### 2.1 本次要解决的问题

本次只解决 `复执行` 这一组能力的策略表达和模板组织问题，具体包括：

1. 用下拉框替换当前 `步` 复选框。
2. 让 `复执行` 根据下拉框当前选项，复制不同的执行提示词。
3. 保留现有“AI 判断执行”的能力。
4. 保留现有“直接执行”的能力。
5. 保留现有“生成步骤文档后停下”的能力。
6. 新增“生成步骤目录后停下”的能力。
7. 将执行类模板拆分到模板子目录，避免继续堆在模板根目录。

### 2.2 本次不解决的问题

本次不处理以下内容：

1. 不改 `建需求`、`复需求`、`建回复`、`复回复`、`复关系` 的业务逻辑。
2. 不改 `实` 复选框的逻辑。
3. 不改窗口整体视觉风格，不做 UI 重设计。
4. 不引入复杂配置化策略系统，不新增策略持久化设置。
5. 不处理正式代码执行链路本身是否成功，只处理“复制什么提示词”的策略选择问题。

### 2.3 必须保持不变的约束

1. 保持当前 AHK 小面板的紧凑布局思路。
2. 保持当前“配置读取 + GUI 创建 + 事件函数 + 模板文件”的代码组织风格。
3. 提示词内容尽量放模板文件，不在主脚本中硬编码大段文本。
4. 用户本次选择的执行策略不回写 `settings.ini`。
5. 程序启动后的默认执行策略固定为 `直接执行`。

## 3. 策略模型

### 3.1 最终策略结果

`复执行` 最终支持 4 种可选策略：

1. `直接执行`
   基于当前目录的 `实施文档.md`，直接修改正式代码和文件，完成方案落地。

2. `AI判断`
   保留当前已有语义。
   AI 只允许在以下两种结果之间判断：
   - 直接执行
   - 生成步骤文档并停下

3. `步骤文档`
   在当前目录下生成多个步骤 md 文档，覆盖实施文档的全部内容，写完后停下等待确认，不直接修改正式代码。

4. `步骤目录`
   在当前目录下生成多个步骤目录，每个目录内放入同名 md 文档；生成后停下等待确认，不直接修改正式代码。

### 3.2 关于第 4 种策略的结构定义

第 4 种策略不是独立命名体系，而是第 3 种策略的结构升级版。

两者关系如下：

1. 第 3 种：
   - `01-xxx.md`
   - `02-xxx.md`
   - `03-xxx.md`

2. 第 4 种：
   - `01-xxx\01-xxx.md`
   - `02-xxx\02-xxx.md`
   - `03-xxx\03-xxx.md`

也就是说：

1. 编号规则保持一致。
2. 命名规则保持一致。
3. 唯一变化是承载层级从“文件”升级为“目录 + 同名文件”。

### 3.3 默认策略

默认策略固定为 `直接执行`。

含义如下：

1. 程序启动后，下拉框初始选中 `直接执行`。
2. 用户如果不主动切换策略，点击 `复执行` 时就复制直接执行模板。
3. 该默认值只影响运行时初始状态，不写回配置文件。

## 4. 界面方案

### 4.1 控件替换方案

当前结构：

1. `复执行` 按钮。
2. `步` 复选框。

目标结构：

1. 保留 `复执行` 按钮。
2. 删除 `步` 复选框。
3. 在原复选框所在区域改为一个执行策略下拉框。

### 4.2 控件布局要求

由于当前窗口宽度只有 `210`，界面空间非常紧，策略选择控件必须服从现有布局密度。

因此：

1. 不扩大成复杂面板。
2. 不在当前版本引入多行说明文本。
3. 下拉框显示文案优先使用短标签或短词。
4. 真实策略语义通过模板内容来完整表达。

### 4.3 选项文案原则

由于面板宽度有限，选项显示不要求在 UI 上完整解释，只要求：

1. 稳定。
2. 易扫读。
3. 能与当前按钮风格匹配。

建议的显示文案方向：

1. `直执`
2. `AI判`
3. `步文`
4. `步目`

或更接近可读性的短词版本：

1. `直接`
2. `AI判`
3. `步骤文`
4. `步骤目`

最终文案可在实施时根据 AHK 下拉框实际宽度微调，但必须遵循“短标签优先”的原则。

### 4.4 用户交互语义

最终用户交互应为：

1. 用户先设置当前主题目录。
2. 用户根据这次任务的执行意图，在 `复执行` 旁的下拉框中选择策略。
3. 点击 `复执行`。
4. 程序根据当前策略，复制对应模板内容到剪贴板。
5. 若当前目录下不存在 `实施文档.md`，则仍沿用现有错误提示，不复制任何内容。

## 5. 模板方案

### 5.1 模板组织目标

执行类模板已经从“2 个模板 + 一个布尔开关”升级为“多模板 + 显式策略选择”。

因此必须将执行类模板拆到子目录管理，避免模板根目录继续扁平堆积。

### 5.2 建议目录结构

在 `app/templates` 下新增执行策略子目录，例如：

```text
app/templates/execute/
  direct.txt
  ai_judge.txt
  steps_file.txt
  steps_dir.txt
```

现有模板根目录下其他文件保持不变，例如：

```text
app/templates/
  requirement_prompt.txt
  reply_prompt.txt
  reply_prompt_impl_tail.txt
  context_relation.txt
  no_modify_prompt.txt
  execute/
    direct.txt
    ai_judge.txt
    steps_file.txt
    steps_dir.txt
```

### 5.3 各模板职责

1. `direct.txt`
   用于“直接执行”。

2. `ai_judge.txt`
   用于“AI判断”。
   语义必须与当前 `execute_step_prompt.txt` 的已有能力一致，即：
   - AI 先判断是否需要拆步骤。
   - 不需要则直接执行。
   - 需要则生成步骤文档并停下。
   - 不进入步骤目录模式。

3. `steps_file.txt`
   用于“步骤文档”。
   明确要求：
   - 在当前目录生成多个步骤 md 文件。
   - 步骤应覆盖实施文档全部内容。
   - 生成后停下等待确认。
   - 不直接修改正式代码。

4. `steps_dir.txt`
   用于“步骤目录”。
   明确要求：
   - 在当前目录生成多个步骤目录。
   - 每个目录中生成与目录同名的 md 文档。
   - 步骤结构和编号规则与步骤文档模式一致。
   - 生成后停下等待确认。
   - 不直接修改正式代码。

### 5.4 模板占位符

继续沿用当前模板替换思路，执行类模板只需要一个核心占位符：

1. `{{filePath}}`
   指向当前目录下的 `实施文档.md`。

这样可以保持现有实现模式一致，避免引入额外模板引擎复杂度。

## 6. 代码改造方案

### 6.1 需要修改的主要文件

本次正式实施时，预计修改以下文件：

1. `C:\NotSystem\ClaudeCodeTool\AIProcess\app\AIProcess.ahk`
2. `C:\NotSystem\ClaudeCodeTool\AIProcess\app\templates\execute\*.txt`

并移除或废弃旧执行模板文件的直接引用关系：

1. `execute_prompt.txt`
2. `execute_step_prompt.txt`

是否保留这两个旧文件本身，可在实施时决定：

1. 如果项目希望减少无用文件，应删除。
2. 如果项目希望降低一次性改动风险，可先不引用，后续再清理。

由于用户已经明确“不留无用代码”，最终目标应是：

1. 不保留旧 `步` 复选框相关逻辑。
2. 不保留旧执行双模板的无效分支。

### 6.2 `AIProcess.ahk` 改造点

#### 6.2.1 全局变量层

当前有：

1. `ExecuteStepCheckbox`

需要改为类似：

1. `ExecuteStrategyDropdown`

如果需要清晰表达策略值，可新增稳定标识，例如：

1. 执行策略常量。
2. 或策略键与模板文件名映射。

目标是避免在点击事件里写一长串硬编码判断。

#### 6.2.2 默认模板初始化

当前 `EnsureDefaultFiles()` 中会保证存在：

1. `execute_prompt.txt`
2. `execute_step_prompt.txt`

实施后应改为保证存在执行子目录和 4 个策略模板文件。

建议做法：

1. 创建 `TemplateDir "\execute"`。
2. 分别校验并创建：
   - `direct.txt`
   - `ai_judge.txt`
   - `steps_file.txt`
   - `steps_dir.txt`

这样能保持“首次运行自动补齐默认模板”的现有项目习惯。

#### 6.2.3 GUI 创建层

当前在 `CreateMainGui()` 中：

1. 创建 `复执行` 按钮。
2. 创建 `步` 复选框。

实施后应改为：

1. 保留 `复执行` 按钮。
2. 在原区域创建下拉框。
3. 默认选中 `直接执行`。

并保证：

1. 宽度适配当前窗口。
2. 启用/禁用状态继续受 `SetControlsEnabled()` 管理。

#### 6.2.4 启用状态控制

当前 `SetControlsEnabled()` 会处理：

1. `CopyExecuteButton`
2. `ExecuteStepCheckbox`

实施后应改为处理：

1. `CopyExecuteButton`
2. `ExecuteStrategyDropdown`

确保未设置当前目录前，该下拉框与其他执行控件一样处于不可用状态。

#### 6.2.5 `CopyExecutePrompt()` 逻辑重构

当前逻辑是：

1. 检查 `实施文档.md` 是否存在。
2. 读取复选框状态。
3. 在 2 个模板间二选一。
4. 替换 `{{filePath}}`。
5. 复制到剪贴板。

实施后应改为：

1. 检查 `实施文档.md` 是否存在。
2. 读取下拉框当前策略。
3. 根据策略映射到对应模板文件。
4. 读取模板内容。
5. 替换 `{{filePath}}`。
6. 复制到剪贴板。
7. 给出与策略对应的反馈消息。

建议将“策略 -> 模板文件 -> 提示消息”的关系抽成一个独立辅助函数或映射表，而不是散落在事件函数内。

#### 6.2.6 清理旧逻辑

正式实施时必须清理以下旧结构：

1. `ExecuteStepCheckbox` 全局变量。
2. 旧复选框 GUI 创建代码。
3. `SetControlsEnabled()` 中对旧复选框的启用控制。
4. `CopyExecutePrompt()` 中基于布尔勾选的模板分支。
5. `EnsureDefaultFiles()` 中对旧执行模板的默认创建逻辑。

目标是做到：

1. 代码里只保留新的策略选择结构。
2. 不留下已废弃但仍可运行的旧路径。

## 7. 提示词内容要求

### 7.1 直接执行模板

必须明确：

1. 阅读当前实施文档。
2. 修改正式代码和文件。
3. 完成整个方案落地。

### 7.2 AI 判断模板

必须保持当前已有能力语义：

1. 先判断实施文档复杂度。
2. 如果不需要拆步骤，则直接修改正式代码和文件。
3. 如果需要拆步骤，则在当前目录创建多个步骤 md 文档。
4. 步骤文档覆盖实施文档全部内容。
5. 步骤文档生成后停下等待验收。
6. 不要自行进入“步骤目录”模式。

### 7.3 步骤文档模板

必须明确：

1. 基于当前实施文档拆解步骤。
2. 在当前目录创建多个步骤 md 文件。
3. 文件按顺序编号。
4. 全部步骤合起来覆盖实施文档全部内容。
5. 步骤文档生成后停下。
6. 未获确认前不修改正式代码。

### 7.4 步骤目录模板

必须明确：

1. 基于当前实施文档拆解步骤。
2. 在当前目录创建多个步骤目录。
3. 目录名按顺序编号。
4. 每个目录中创建一个与目录同名的 md 文档。
5. 这些目录和文档合起来覆盖实施文档全部内容。
6. 生成后停下。
7. 未获确认前不修改正式代码。

## 8. 验收标准

正式实施完成后，至少应满足以下验收点。

### 8.1 界面行为验收

1. 主界面中 `步` 复选框已消失。
2. `复执行` 按钮旁出现执行策略下拉框。
3. 下拉框默认选中 `直接执行` 对应选项。
4. 未设置当前目录时，下拉框与 `复执行` 按钮均不可用。
5. 设置当前目录后，下拉框与 `复执行` 按钮恢复可用。

### 8.2 模板行为验收

1. 选择 `直接执行` 时，复制的是直接执行模板。
2. 选择 `AI判断` 时，复制的是 AI 判断模板。
3. 选择 `步骤文档` 时，复制的是步骤文档模板。
4. 选择 `步骤目录` 时，复制的是步骤目录模板。
5. 四种模板都能正确替换 `{{filePath}}` 为当前目录下的 `实施文档.md`。

### 8.3 文件与结构验收

1. `app/templates` 下存在执行模板子目录。
2. 执行模板子目录内存在 4 个策略模板文件。
3. 旧的 `步` 相关代码已清理，不再参与运行。
4. 不存在仅为兼容旧逻辑而残留的无效分支。

### 8.4 使用体验验收

1. 用户可以在一个入口下清楚切换执行策略。
2. 新增的“步骤目录”策略可以直接替代当前手工补救流程。
3. 对现有其他按钮行为没有引入回归影响。

## 9. 实施顺序建议

正式进入代码实施阶段时，建议按下面顺序进行：

1. 先改模板结构。
   创建执行子目录和 4 个模板，保证模板语义先稳定。

2. 再改 `EnsureDefaultFiles()`。
   让首次运行默认文件创建逻辑与新模板结构一致。

3. 再改 GUI。
   用下拉框替换旧复选框，并设置默认值。

4. 再改 `CopyExecutePrompt()`。
   将执行复制逻辑从“布尔分支”重构为“策略映射”。

5. 最后清理旧代码。
   删除 `ExecuteStepCheckbox` 及相关旧分支，确保不留无用代码。

6. 做回归验证。
   重点验证：
   - `复执行` 四种策略
   - `禁改正式` 不受影响
   - `实` 复选框不受影响
   - 其他按钮无回归

## 10. 结论

本次需求的本质，不是简单新增一个控件，而是把 `复执行` 从“按钮 + 二元布尔开关”的旧模型，升级为“单入口 + 多策略显式选择”的新模型。

该升级必须满足三个核心目标：

1. 保留当前已存在且有效的执行能力。
2. 直接支持用户已在真实使用中的“步骤目录”工作流。
3. 继续保持项目当前的 UI 风格、代码风格和模板可维护性。

只要按本文档中的边界和技术方案实施，需求即可完整落地，且不会把当前项目推向过度设计。
