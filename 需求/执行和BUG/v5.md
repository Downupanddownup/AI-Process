# v5

这轮你的回复已经把大方向基本定下来了，所以这版我不再继续讨论“要不要这样做”，而是把我现在理解到的最终方案骨架写清楚，并补几个实施前最好先钉死的小点。

## 我现在确认下来的方案方向

### 1. `复执行` 旁边的旧复选框直接替换为下拉框

这里已经明确了：

1. 位置不变，就放在 `复执行` 按钮旁边。
2. 旧的 `步` 复选框不保留。
3. 不做兼容式并存，而是直接替换。

这说明后续界面仍然保持当前“小面板 + 紧凑操作区”的风格，只是把原来的二元开关升级成可表达多策略的选择器。

### 2. 默认策略是 `直接执行`

这个默认值很重要，它实际上定义了你最常见的使用路径。

也就是说：

1. 用户什么都不改时，点击 `复执行`，复制出来的是“直接执行”的提示词。
2. 只有在用户明确要其他策略时，才主动切换下拉框。

这符合当前工具一贯的“尽量少操作、尽量快触发”的设计思路。

### 3. 模板需要拆到子目录管理

这一点我认同，而且这是必要的。

因为 `复执行` 现在已经不再是一个模板 + 一个布尔分支能解决的问题了，后面至少会有多个稳定策略模板。把它们拆到模板子目录里，会比继续堆在根目录更清楚。

### 4. 改动方式要保持现有项目风格

这点我理解成三个约束：

1. UI 仍然保持现在这种朴素、紧凑、直接的桌面工具风格。
2. 代码组织仍然走当前这种“配置读取 + GUI 创建 + 事件函数 + 模板文件”的结构，不搞大幅重构。
3. 模板内容尽量可独立修改，不把复杂提示词硬编码进主脚本。

## 我现在认为已经基本成立的策略集合

如果按我们前几轮已经对齐的模型，再结合你这轮的默认值要求，我现在认为最终下拉框最合理的一版，至少应当覆盖这 4 个策略：

1. `直接执行`
   复制“基于实施文档直接修改正式代码和文件”的提示词。

2. `AI判断执行`
   复制“由 AI 在直接执行 / 生成步骤文档之间判断”的提示词。

3. `生成步骤文档`
   复制“在当前目录生成多个步骤文档，写完后停下等待确认”的提示词。

4. `生成步骤目录`
   复制“在当前目录生成多个步骤目录，并在目录内生成同名 md 文档，写完后停下等待确认”的提示词。

这里我特意把 `直接执行` 放在第一位，因为你已经明确要求它做默认值。

## 我现在更倾向的模板组织方式

你说可以在模板目录下建子目录，我认为这是对的。按当前项目风格，我会更倾向类似下面这种思路：

1. 保留现有通用模板目录结构。
2. 在其中新增一个专门给执行策略使用的子目录。
3. 每种策略单独一个模板文件。

我之所以倾向“一策略一模板”，而不是“一个模板里用很多占位符和条件拼接”，原因很简单：

1. 后续更容易单独修改文案。
2. 以后再加策略时，不容易把已有逻辑搞乱。
3. 对非代码维护者也更直观。

## 我现在还剩下的几个小确认点

这些已经不是方向性问题了，而是为了避免后面落地时出现歧义。

### 1. 下拉框显示文案，是偏“短标签”还是“完整标签”

因为主界面空间比较紧。

我当前能想到两种风格：

1. 短标签
   - 直执
   - AI判
   - 步文
   - 步目

2. 完整标签
   - 直接执行
   - AI判断
   - 步骤文档
   - 步骤目录

从可读性上我更偏向完整标签，但要看当前窗口宽度是否够用，以及是否需要顺带微调这一区域布局。

### 2. `AI判断执行` 的文案是否要强调“只在 1/2 之间判断”

因为如果只是写 `AI判断`，用户后续可能会误以为 AI 也会判断是否走第 3 种。

所以这里后面可能要考虑：

1. 界面文案尽量短。
2. 真实限制在模板语义里写清楚。

也就是说，UI 上可以简单，但提示词里要明确：

`只允许在直接执行和生成步骤文档之间判断，不要进入步骤目录策略。`

### 3. 第 4 种策略的输出目录命名规则，是否需要现在就写死

你前面已经定义了：

1. 先创建步骤目录。
2. 目录名与其中的步骤 md 文档同名。

这里还剩一个很细的小点：

步骤序号格式要不要保持和第二种一致，例如：

1. `01-xxx\01-xxx.md`
2. `02-xxx\02-xxx.md`

如果这个规则准备和第二种统一，那后面模板和实现描述会更稳定。

### 4. 默认值是否只影响 UI 初始显示，还是也要写回配置

你这轮说了“默认是直接执行”，我目前先理解为：

1. 程序启动时默认选中 `直接执行`。

但还有一个实现层小点：

1. 用户本次改成了 `生成步骤目录`。
2. 下次打开程序时，是恢复上次选择？
3. 还是永远回到 `直接执行`？

如果不特别要求，我从当前工具风格判断，更像是“每次启动都回到默认值”，而不是持久化这个策略选择。

## 我当前的总体判断

我认为需求现在已经接近收敛了。

真正需要落地的核心点，其实只有四个：

1. 把 `步` 复选框升级成策略下拉框。
2. 让 `复执行` 根据下拉框复制不同模板。
3. 把执行类模板拆到子目录中单独管理。
4. 整个改动保持现有 AHK 项目的 UI 和代码组织风格。

如果你认可，这一轮之后我觉得基本就可以进入正式实施文档阶段了；除非你还想先把上面那几个小点再钉一下。
