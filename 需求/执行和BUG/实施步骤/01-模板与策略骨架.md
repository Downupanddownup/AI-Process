# 01-模板与策略骨架

## 目标

先完成执行策略的数据骨架和模板骨架调整，为后续 GUI 和事件逻辑改造提供稳定基础。

本步骤只处理“执行策略如何被程序表达”和“模板文件如何组织”，不处理界面控件替换和最终交互联动。

## 范围

1. 在 `app/templates` 下新增执行模板子目录。
2. 建立 4 种执行策略对应的默认模板文件。
3. 在主脚本中建立执行策略的稳定标识、显示文案、模板文件映射、反馈文案映射。
4. 调整默认模板初始化逻辑，使首次运行时可以自动补齐新模板结构。

## 具体实施内容

### 1. 新增执行模板子目录

目标目录：

```text
app/templates/execute/
```

目录内模板文件：

1. `direct.txt`
2. `ai_judge.txt`
3. `steps_file.txt`
4. `steps_dir.txt`

## 2. 迁移执行类模板职责

将现有：

1. `execute_prompt.txt`
2. `execute_step_prompt.txt`

迁移为 4 个策略模板：

1. `direct.txt`
   对应“直接执行”

2. `ai_judge.txt`
   对应“AI判断”

3. `steps_file.txt`
   对应“步骤文档”

4. `steps_dir.txt`
   对应“步骤目录”

其中：

1. `direct.txt` 可继承当前 `execute_prompt.txt` 的核心语义。
2. `ai_judge.txt` 可继承当前 `execute_step_prompt.txt` 的核心语义，但要明确只在“直接执行 / 步骤文档”之间判断。
3. `steps_file.txt` 和 `steps_dir.txt` 需要分别固化为强制策略模板。

## 3. 在主脚本中建立策略骨架

建议引入稳定的策略键，例如：

1. `direct`
2. `ai_judge`
3. `steps_file`
4. `steps_dir`

并建立以下映射关系：

1. 策略键 -> 下拉框显示文案
2. 策略键 -> 模板文件路径
3. 策略键 -> 复制成功提示文案

目标是让后续 `CopyExecutePrompt()` 不再依赖布尔勾选，而是依赖策略映射。

## 4. 调整默认模板初始化逻辑

需要改造 `EnsureDefaultFiles()`，使其：

1. 创建 `app/templates/execute/` 目录。
2. 确保 4 个执行策略模板存在。
3. 不再把旧的“执行双模板 + 步复选框”作为唯一结构。

## 产出结果

完成本步骤后，应具备以下基础能力：

1. 程序内部已经有明确的执行策略标识。
2. 执行模板已经按策略拆分到子目录。
3. 后续 GUI 下拉框和执行事件逻辑可以直接复用这些策略定义。

## 验收点

1. `app/templates/execute/` 已创建。
2. 4 个执行模板文件已存在且内容语义正确。
3. `EnsureDefaultFiles()` 已能自动补齐新执行模板结构。
4. 主脚本中已存在稳定的执行策略映射，不再依赖旧 `步` 布尔值作为唯一策略来源。
